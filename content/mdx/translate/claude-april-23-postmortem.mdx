> 원문: [An update on recent Claude Code quality reports](https://www.anthropic.com/engineering/april-23-postmortem) — Anthropic Engineering

지난 한 달 동안 일부 사용자에게 Claude의 응답 품질이 저하되었다는 보고를 검토해 왔습니다. 이러한 보고는 Claude Code, Claude Agent SDK, Claude Cowork에 영향을 미친 세 가지 별도의 변경 사항으로 추적되었습니다. API는 영향을 받지 않았습니다.

세 가지 문제 모두 4월 20일(v2.1.116)부로 해결되었습니다.

이 게시물에서는 우리가 발견한 내용, 수정된 내용, 그리고 유사한 문제가 다시 발생할 가능성을 크게 줄이기 위해 다르게 수행할 작업에 대해 설명합니다.

저희는 품질 저하에 대한 보고를 매우 심각하게 받아들입니다. 저희는 모델의 성능을 의도적으로 저하시키지 않으며, API와 추론 계층에는 영향이 없음을 즉시 확인할 수 있었습니다.

조사 후, 세 가지 다른 문제를 확인했습니다.

1.  3월 4일, 일부 사용자에게 `high` 모드에서 UI가 멈춘 것처럼 보이게 할 정도로 매우 긴 지연 시간을 줄이기 위해 Claude Code의 기본 추론 노력을 `high`에서 `medium`으로 변경했습니다. 이는 잘못된 절충이었습니다. 사용자들이 단순 작업에는 낮은 노력을 선택하고 높은 지능을 기본으로 선호한다고 알려왔기 때문에 4월 7일 이 변경 사항을 되돌렸습니다. 이는 Sonnet 4.6 및 Opus 4.6에 영향을 미쳤습니다.
2.  3월 26일, 1시간 이상 유휴 상태였던 세션에서 Claude의 이전 추론을 지워 세션을 재개할 때 지연 시간을 줄이는 변경 사항을 배포했습니다. 버그로 인해 이 작업이 세션의 나머지 부분에 대해 한 번이 아니라 매 턴마다 계속 발생하여 Claude가 건망증이 있고 반복적인 것처럼 보이게 만들었습니다. 4월 10일에 이를 수정했습니다. 이는 Sonnet 4.6 및 Opus 4.6에 영향을 미쳤습니다.
3.  4월 16일, 장황함을 줄이기 위해 시스템 프롬프트 지침을 추가했습니다. 다른 프롬프트 변경 사항과 결합되어 코딩 품질을 저하시켰으며 4월 20일에 되돌렸습니다. 이는 Sonnet 4.6, Opus 4.6, Opus 4.7에 영향을 미쳤습니다.

각 변경 사항이 서로 다른 트래픽의 다른 부분을 다른 일정으로 영향을 미쳤기 때문에 집계 효과는 광범위하고 일관성 없는 저하처럼 보였습니다. 3월 초에 보고에 대한 조사를 시작했지만, 처음에는 정상적인 사용자 피드백 변동과 구별하기 어려웠으며, 내부 사용 또는 평가 모두 처음에 식별된 문제를 재현하지 못했습니다.

이는 사용자가 Claude Code에서 기대해야 하는 경험이 아닙니다. 4월 23일부로 모든 구독자에게 사용량 제한을 재설정합니다.

# Claude Code의 기본 추론 노력 변경

2월에 Claude Code에서 Opus 4.6을 출시했을 때, 기본 추론 노력을 `high`로 설정했습니다.

곧이어 Opus 4.6에서 Claude가 높은 노력 모드에서 가끔 너무 오래 생각하여 UI가 멈춘 것처럼 보이고 해당 사용자에게 불균형적인 지연 및 토큰 사용량이 발생하는 것에 대한 사용자 피드백을 받았습니다.

일반적으로 모델이 더 오래 생각할수록 결과가 더 좋습니다. 노력 수준은 Claude Code가 사용자에게 해당 절충을 설정할 수 있도록 하는 방법입니다. 즉, 더 많은 사고 대 낮은 지연 시간 및 적은 사용량 제한 발생입니다. 모델의 노력 수준을 조정할 때, 테스트 시간-컴퓨팅 곡선을 따라 사람들이 최상의 옵션 범위를 제공하는 지점을 선택하기 위해 이 절충을 고려합니다. 제품 계층에서는 이 곡선을 따라 기본값으로 설정할 지점을 선택하고, 이를 Messages API에 노력 매개변수로 보냅니다. 그런 다음 `/effort`를 통해 다른 옵션을 제공합니다.

![](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2Fde3bcf9733b61f57234d8c45e663b1bd48677ea1-3840x2160.png&w=3840&q=75)

내부 평가 및 테스트에서, 대부분의 작업에 대해 중간 노력이 상당히 적은 지연 시간으로 약간 낮은 지능을 달성했습니다. 또한 사고에 대한 매우 긴 꼬리 지연 시간의 동일한 문제로 고통받지 않았으며, 사용자의 사용량 제한을 최대화하는 데 도움이 되었습니다. 결과적으로, 중간 노력을 기본 노력으로 만드는 변경 사항을 배포했으며 제품 내 대화를 통해 그 이유를 설명했습니다.

![](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2F459b2a8a0baa88937eebcbe4566dde4d6cc7f185-3794x2260.png&w=3840&q=75)

배포 직후, 사용자들은 Claude Code가 덜 똑똑하다고 보고하기 시작했습니다. 기본 설정을 더 명확하게 만들기 위해 여러 디자인 반복을 배포하여 사람들이 기본값을 변경할 수 있다는 것을 알렸습니다(시작 시 알림, 인라인 노력 선택기, ultrathink 복원). 그러나 대부분의 사용자는 중간 노력 기본값을 유지했습니다.

더 많은 고객으로부터 피드백을 받은 후, 4월 7일에 이 결정을 번복했습니다. 이제 모든 사용자는 Opus 4.7의 경우 `xhigh` 노력, 다른 모든 모델의 경우 `high` 노력으로 기본 설정됩니다.

# 이전 추론을 삭제한 캐싱 최적화

Claude가 작업을 추론할 때, 해당 추론은 일반적으로 대화 기록에 유지되어 후속 턴마다 Claude가 왜 해당 편집 및 도구 호출을 했는지 볼 수 있습니다.

3월 26일, 이 기능에 대한 효율성 개선이 될 예정이었던 변경 사항을 배포했습니다. 사용자에게 백투백 API 호출을 더 저렴하고 빠르게 만들기 위해 프롬프트 캐싱을 사용합니다. Claude는 API 요청 시 입력 토큰을 캐시에 기록한 다음, 일정 시간 동안 비활성 상태가 지나면 프롬프트가 캐시에서 제거되어 다른 프롬프트를 위한 공간을 확보합니다. 캐시 활용도는 신중하게 관리하는 사항입니다(저희의 [접근 방식](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything)에 대해 자세히 설명).

디자인은 간단해야 했습니다. 세션이 1시간 이상 유휴 상태였다면, 오래된 사고 섹션을 지워 세션 재개 비용을 줄일 수 있었습니다. 요청은 어차피 캐시 미스가 될 것이므로, API로 전송되는 캐시되지 않은 토큰 수를 줄이기 위해 요청에서 불필요한 메시지를 제거할 수 있었습니다. 그런 다음 전체 추론 기록을 다시 보내기 시작했습니다. 이 작업을 수행하기 위해 `clear_thinking_20251015` API 헤더와 `keep:1`을 사용했습니다.

구현에 버그가 있었습니다. 사고 기록을 한 번 지우는 대신, 세션의 나머지 부분의 매 턴마다 지웠습니다. 세션이 한 번 유휴 임계값을 넘으면, 해당 프로세스의 나머지 부분에 대한 각 요청은 가장 최근의 추론 블록만 유지하고 이전의 모든 것을 폐기하도록 API에 지시했습니다. 이것이 누적되었습니다. Claude가 도구 사용 중간에 후속 메시지를 보낸 경우, 이는 잘못된 플래그 아래 새 턴을 시작했으므로 현재 턴의 추론조차 삭제되었습니다. Claude는 계속 실행되었지만, 왜 그렇게 하기로 선택했는지에 대한 기억이 점점 줄어들었습니다. 이것은 사람들이 보고한 건망증, 반복, 이상한 도구 선택으로 나타났습니다.

후속 요청에서 이러한 사고 블록이 지속적으로 삭제되었기 때문에 해당 요청은 캐시 미스를 유발했습니다. 이것이 예상보다 빠르게 사용량 제한이 소진되었다는 별도의 보고를 유발했다고 생각합니다.

![](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2F332d9c487bb73c8078686068dcbe1b616720a8dd-3016x1198.png&w=3840&q=75)

두 개의 관련 없는 실험으로 인해 처음에는 이 문제를 재현하기 어려웠습니다. 메시지 큐와 관련된 내부 전용 서버 측 실험이었고, 사고 표시 방식의 다른 변경 사항으로 인해 대부분의 CLI 세션에서 이 버그가 숨겨져 외부 빌드를 테스트할 때조차 이를 감지하지 못했습니다.

이 버그는 Claude Code의 컨텍스트 관리, Anthropic API, 그리고 확장된 사고의 교차점에 있었습니다. 이로 인해 발생한 변경 사항은 여러 번의 사람 및 자동화된 코드 검토, 단위 테스트, 엔드투엔드 테스트, 자동화된 검증 및 도그푸딩을 통과했습니다. 또한 코너 케이스(오래된 세션)에서만 발생하고 문제를 재현하기 어렵다는 점과 결합하여 근본 원인을 발견하고 확인하는 데 일주일 이상이 걸렸습니다.

조사의 일환으로 Opus 4.7을 사용하여 문제가 되는 풀 요청에 대해 [Code Review](https://code.claude.com/docs/en/code-review)를 백테스트했습니다. Opus 4.7은 완전한 컨텍스트를 수집하는 데 필요한 코드 리포지토리가 제공되었을 때 버그를 발견했지만, Opus 4.6은 발견하지 못했습니다. 이러한 일이 다시 발생하지 않도록, 이제 코드 검토에 대한 컨텍스트로 추가 리포지토리에 대한 지원을 제공합니다.

v2.1.101에서 4월 10일에 이 버그를 수정했습니다.

# 장황함 감소를 위한 시스템 프롬프트 변경

최신 모델인 Claude Opus 4.7은 이전 모델에 비해 눈에 띄는 행동 특성이 있습니다. 출시 시 [글에서 언급했듯이](https://www.anthropic.com/news/claude-opus-4-7), 매우 장황한 경향이 있습니다. 이는 어려운 문제에 대해 더 똑똑하게 만들지만, 더 많은 출력 토큰을 생성합니다.

Opus 4.7을 출시하기 몇 주 전에, 준비를 위해 Claude Code를 튜닝하기 시작했습니다. 각 모델은 약간 다르게 작동하며, 각 릴리스 전에 하네스와 제품을 최적화하는 데 시간을 보냅니다.

장황함을 줄이기 위한 여러 도구가 있습니다. 모델 훈련, 프롬프팅, 제품 내 사고 UX 개선입니다. 궁극적으로 이 모든 것을 사용했지만, Claude Code의 지능에 상당한 영향을 미친 시스템 프롬프트에 대한 한 가지 추가 사항은 다음과 같습니다.

> _“길이 제한: 도구 호출 사이의 텍스트는 25단어 이하로 유지하세요. 작업에 더 자세한 내용이 필요한 경우가 아닌 한 최종 응답은 100단어 이하로 유지하세요.”_

몇 주간의 내부 테스트와 실행한 평가 세트에서 회귀가 발생하지 않은 후, 변경 사항에 대해 확신을 갖고 4월 16일 Opus 4.7과 함께 배포했습니다.

이 조사의 일환으로, 더 넓은 평가 세트를 사용하여 각 줄의 영향을 이해하기 위해 시스템 프롬프트에서 줄을 제거하는 방식으로 더 많은 어블레이션(ablations)을 실행했습니다. 이러한 평가 중 하나는 Opus 4.6과 4.7 모두에서 3%의 성능 저하를 보여주었습니다. 4월 20일 릴리스의 일부로 즉시 프롬프트를 되돌렸습니다.

# 향후 계획

이러한 문제를 피하기 위해 몇 가지 사항을 다르게 수행할 것입니다. 내부 직원의 더 많은 비율이 퍼블릭 빌드와 정확히 동일한 Claude Code 빌드를 사용하도록 할 것입니다(새 기능을 테스트하는 데 사용하는 버전이 아닌). 그리고 내부적으로 사용하는 [Code Review](https://code.claude.com/docs/en/code-review) 도구를 개선하여 고객에게 개선된 버전을 제공할 것입니다.

또한 시스템 프롬프트 변경에 대한 더 엄격한 제어를 추가하고 있습니다. Claude Code의 모든 시스템 프롬프트 변경에 대해 모델별 평가를 광범위하게 실행하고, 각 줄의 영향을 이해하기 위해 어블레이션을 계속 수행하며, 프롬프트 변경을 더 쉽게 검토하고 감사할 수 있는 새로운 도구를 구축했습니다. 또한 모델별 변경 사항이 대상으로 하는 특정 모델에 국한되도록 CLAUDE.md에 대한 지침을 추가했습니다. 지능과 절충될 수 있는 모든 변경 사항에 대해, 더 일찍 문제를 발견할 수 있도록 충분한 기간, 더 광범위한 평가 세트, 점진적인 롤아웃을 추가할 것입니다.

최근에 제품 결정과 그 이면에 있는 이유를 심층적으로 설명할 수 있는 공간을 확보하기 위해 X에 @ClaudeDevs를 생성했습니다. GitHub의 중앙 집중식 스레드에서도 동일한 업데이트를 공유할 것입니다.

마지막으로 사용자들에게 감사드립니다. `/feedback` 명령을 사용하여 문제를 공유해 주신 분들(또는 온라인에 구체적이고 재현 가능한 예시를 게시하신 분들)이 궁극적으로 이러한 문제를 식별하고 수정할 수 있도록 했습니다. 오늘 모든 구독자에게 사용량 제한을 재설정합니다.
