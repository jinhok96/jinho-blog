_당사의 시리즈_ **_스타트업, Claude로 구축하기_**_에서는 스타트업이 AI로 산업을 혁신하는 방법을 조명합니다. 이 글에서는 CodeRabbit이 AI가 코드를 생성하기 전에 계획하는 에이전트 오케스트레이션 계층을 구축한 방법을 공유합니다.

AI 코딩 도구는 아이디어와 작동하는 프로토타입 사이의 시간을 단축했습니다. AI 코드 검토 플랫폼인 CodeRabbit은 처리량과 함께 증가하는 또 다른 추세를 발견했습니다. 바로 컴파일되고 테스트를 통과하지만 팀이 실제로 구축하려 했던 것을 수행하지 못하는 코드입니다.

CodeRabbit의 AI 부문 VP인 David Loker는 원인을 모델의 상류에서 찾습니다. 경험이 풍부한 개발자는 종종 코딩 에이전트가 자신과 동일한 맥락을 이해한다고 가정하므로, 자신에게는 분명해 보이는 요구 사항을 작성하지 않습니다. 그런 다음 코딩 에이전트는 자신이 합리적이라고 생각하는 것으로 격차를 채웁니다.

이 격차를 해소하기 위해 CodeRabbit은 Claude를 사용하여 코드가 생성되기 전에 구조화된 계획 단계를 실행하는 에이전트 오케스트레이션 시스템을 설계하고 구축했습니다. 팀의 작동 가설은 계획의 품질이 결과의 품질을 결정하며, 코드 생성 비용이 저렴해질수록 잘못된 방향으로 이동하는 비용이 더 많이 든다는 것입니다.

# **AI 코딩의 내부 지식 격차 해결**

CodeRabbit 팀이 고객 기반 전체의 AI 생성 풀 요청을 연구했을 때, 가장 빈번한 실패 모드는 컴파일되고 테스트를 통과했지만 문제를 해결하지 못한 코드였습니다.

Loker는 "개발자로서 경험을 쌓으면서 지식을 내면화합니다. 이 모든 것들이 머릿속에 있고 다른 개발자들도 그것을 알고 있다고 가정합니다. 하지만 AI 시스템에도 암묵적으로 이해한다고 가정합니다. 이러한 것들을 가정하고 있다는 사실조차 인식하지 못합니다."라고 말합니다.

모호한 프롬프트는 기본 시스템이 자신이 합리적이라고 생각하는 것으로 격차를 채우도록 강요합니다. 이 추측은 종종 개발자가 생각했던 것과 달라집니다.

Loker는 개인적인 예를 제시합니다. 사이드 프로젝트에서 메모리 시스템을 구축하는 동안 그는 모든 것이 실행될 때까지 코딩 에이전트와 몇 시간 동안 반복했습니다. 에이전트에게 사용 방법을 묻자 지침은 사용자 토큰을 전달하라고 했습니다. 로그인 페이지가 없었습니다. 그는 시스템이 사용자를 요구한다고 명시했지만 사용자가 로그인할 방법이 필요하다고는 말하지 않았습니다. 에이전트는 격차를 채웠고, 몇 시간의 작업은 정문이 없는 제품으로 귀결되었습니다.

Loker는 "결국 그 위에 더 많은 것을 구축하게 되고, 훨씬 나중에 문제가 있다는 것을 발견하게 됩니다. AI 워크플로우에서는 후기 검증이 매우 비쌀 수 있습니다."라고 말합니다.

# **AI 코딩 솔루션 이전에 실행되는 오케스트레이션 계층**

CodeRabbit의 대응은 코드 생성 앞에 계획 시스템을 삽입하는 것이었습니다. 이 시스템은 여러 Claude 모델을 조정하여 요구 사항을 분석하고 코드 생성 전에 가정을 드러내어 무엇을 구축해야 하는지, 어떤 제약 조건을 충족해야 하는지를 정의하는 구조화된 실행 계획을 생성합니다.

Loker는 "이 계획 시스템은 Claude Code의 Plan Mode를 대체하기 위한 것이 아닙니다. Claude Code 이전에 발생하는 더 높은 수준의 오케스트레이션으로, 명확해야 할 모든 것이 명확해지고 어떤 가정이 이루어지는지 인식할 수 있도록 매우 좁고 올바른 방향으로 안내하는 역할을 합니다."라고 말합니다.

결과는 협업 제품 요구 사항 문서(PRD)입니다. 즉, 전체 맥락으로 생성되고 팀 전체의 이해 관계자가 검증하며 구현이 시작되기 전에 검토되는 계획입니다. Claude Code는 이 계획을 받아 미세한 구현 계획을 생성하는 데 사용합니다. 이 계획은 결정된 내용과 그 이유를 캡처하는 공유 아티팩트가 되어 팀이 재작업을 피하고 결과가 원래 의도와 일치했는지 나중에 검증하는 데 도움이 될 뿐만 아니라 신규 엔지니어의 온보딩도 지원합니다.

# **Claude 모델 제품군 간의 라우팅**

CodeRabbit은 비용과 지연 시간을 최적화하기 위해 각 모델 계층을 작업 복잡성과 일치시킵니다. Opus는 오케스트레이션 루프와 문제 이해 및 전반적인 방향 설정과 같은 더 높은 수준의 전략적 작업을 주도합니다. Sonnet은 해당 출력을 받아 구조화된 계획 단계로 순차화합니다. Haiku는 질문이 더 작은 모델이 잘 답변할 수 있을 만큼 구체적인 컨텍스트 증류 및 대상 도구 사용과 같이 범위가 좁은 작업을 처리합니다.

Loker는 "Haiku가 주어진 작업에서 Sonnet만큼 잘 수행된다면 Haiku를 사용합니다. 평가 하네스가 계획 품질이 Opus에 더 많은 여지를 줄 때 개선된다고 알려준다면 Opus에 더 많은 여지를 줍니다. 추측하지 않습니다."라고 말합니다.

# **계획 품질을 위한 평가 하네스 구축**

CodeRabbit은 코드 검토를 위한 성숙한 평가 시스템을 가지고 있었지만, 계획 결과 평가를 위한 시스템은 없었습니다. 해당 인프라를 구축하는 것은 자체 프로젝트가 되었습니다.

시스템은 수동 조정된 예제와 수동 검사로 시작되었습니다. 팀은 계획 품질의 특정 차원을 점수화하는 LLM 심판 라이브러리를 개발했습니다. 계획은 결국 코드를 생성하므로 팀은 생성된 코드가 작동하는지, 추가 범위가 포함되었는지, 거기에 도달하는 데 몇 개의 토큰이 걸렸는지도 측정할 수 있었습니다. 계획 단계의 유무에 따라 동일한 작업을 실행하면 계획 자체의 가치를 격리할 수 있었습니다.

Loker는 "계획의 올바른 세부 수준이 어느 정도가 될지 예상하지 못했습니다. 계획이 너무 세분화되면 코드베이스가 변경되는 순간 쓸모없게 되었습니다. 계획이 너무 높은 수준이면 에이전트가 가정을 채울 여지가 남았는데, 이는 계획 계층이 해결하려 했던 원래 문제였습니다. 적절한 추상화 수준을 찾는 데는 반복이 필요했으며, 이는 평가 하네스가 가능하게 한 것입니다."라고 말합니다.

# **코드가 작성되기 전에 오류 감지**

AI 네이티브 코딩 워크플로우에서는 코드 검토 중에 발생하던 많은 결정이 이제 계획 계층에서 더 일찍 이루어집니다. 코드 생성이 시작되기 전에 팀이 검토하고 합의할 수 있는 계획을 구축하면 실수를 조기에 감지할 수 있습니다.

Loker는 "Claude 생태계를 사용하여 구축한 것은 팀 전체의 계획 시스템입니다. 계획 자체가 품질 게이트가 됩니다. 초기 단계에서 해당 계획의 품질이 정말 좋다고 확신할 수 있다면 그 결과는 매우 두드러집니다. 결국 더 나은 코드가 생성됩니다."라고 말합니다.
