Claude Code에 오늘 동적 워크플로우를 도입하여 Claude가 가장 어려운 작업을 엔드투엔드로 처리할 수 있도록 지원합니다. 일반적으로 분기별로 계획했던 작업이 이제는 며칠 만에 완료됩니다. Claude는 오케스트레이션 스크립트를 동적으로 작성하여 단일 세션에서 수십에서 수백 개의 병렬 하위 에이전트를 실행하고, 결과물을 사용자에게 전달하기 전에 자체적으로 검증합니다.

단일 에이전트의 한 번의 실행으로 처리하기에는 너무 큰 문제들이 있습니다. 특히 복잡하고 레거시 코드베이스에서는 더욱 그렇습니다. 전체 서비스에 걸친 버그 헌팅, 수백 개의 파일을 건드리는 마이그레이션, 커밋하기 전에 모든 각도에서 스트레스 테스트를 원하는 계획 등이 이에 해당합니다. 동적 워크플로우는 이러한 모든 작업을 엔드투엔드로 처리할 수 있습니다.

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a186b2e070156fbb2df90ad_166befe7.png)

동적 워크플로우는 오늘부터 Claude Code CLI, Desktop, VS Code 확장 프로그램(Max, Team, Enterprise 플랜의 경우 관리자 활성화 시) 및 Claude API, Amazon Bedrock, Vertex AI, Microsoft Foundry에서 연구 미리보기로 제공됩니다.

참고: 동적 워크플로우는 일반적인 Claude Code 세션보다 훨씬 많은 토큰을 소비할 수 있으므로, 사용량을 파악하기 위해 범위가 좁은 작업부터 시작하는 것을 권장합니다.

최상의 경험을 위해 동적 워크플로우를 사용할 때는 자동 모드를 켜십시오. 여기에서 워크플로우를 시작하는 두 가지 방법이 있습니다.

1. Claude에게 직접 동적 워크플로우 생성을 요청합니다 (예: "워크플로우 생성").
2. Claude Code에 새로 추가된 `ultracode`라는 설정을 켭니다. 이는 노력 메뉴를 통해 접근할 수 있으며, 노력 수준을 xhigh로 설정하면서 Claude가 작업 처리에 워크플로우를 사용할 시점을 자동으로 결정하도록 합니다.

# **동적 워크플로우 작동 방식**

## **동적 워크플로우 시연**

얼리 액세스 사용자 및 Anthropic 내부 팀은 다양한 사용 사례에 동적 워크플로우를 사용해 왔으며, 여기에는 다음이 포함됩니다.

*   **코드베이스 전체 버그 헌팅, 프로파일러 기반 최적화 감사 및 보안 감사:** Claude는 서비스 또는 리포지토리를 병렬로 검색한 다음, 모든 발견 사항에 대해 독립적인 검증을 실행하여 보고서에 실제 문제를 표시합니다. 동일한 방식은 강화 작업에도 적용됩니다. 전체 코드베이스에 걸쳐 인증 확인, 입력 유효성 검사 및 안전하지 않은 패턴을 처리합니다.
*   **대규모 마이그레이션 및 현대화 노력:** Claude는 프레임워크 교체, API 폐기, 수천 개의 파일에 걸친 언어 포팅을 엔드투엔드로 처리할 수 있습니다.**‍**
*   **두 번 확인해야 하는 중요한 작업:** 잘못된 결과의 비용이 높은 경우, 워크플로우를 통해 Claude는 문제에 대해 독립적인 시도를 수행하고 결과물을 무효화하려는 적대적인 에이전트와 협력하여 사용자에게 제공합니다.

## **동적 워크플로우를 사용한 Bun 재작성**

동적 워크플로우가 확장성을 제공하는 예시로 최근 Bun의 재작성이 있습니다. Jarred Sumner는 동적 워크플로우를 사용하여 Bun을 Zig에서 Rust로 포팅했으며, 기존 테스트 스위트의 99.8%가 통과하고 약 750,000줄의 Rust 코드가 작성되었으며, 첫 커밋부터 병합까지 11일이 소요되었습니다. 한 워크플로우는 Zig 코드베이스의 각 구조체 필드에 대한 올바른 Rust 수명(lifetime)을 매핑했습니다. 다음 워크플로우는 수백 개의 에이전트가 병렬로 작업하고 각 파일마다 두 명의 검토자가 참여하여 동작적으로 동일한 Zig 파트너의 Rust(.rs) 파일을 작성했습니다. 그런 다음 수정 루프가 빌드 및 테스트 스위트를 실행하여 둘 다 정상적으로 작동할 때까지 반복했습니다. 포팅이 완료된 후, 하룻밤 동안 실행된 워크플로우는 불필요한 데이터 복사를 처리하고 최종 검토를 위해 각 작업에 대한 PR을 열었습니다. 아직 프로덕션 단계는 아니지만, 이 모든 것이 동적 워크플로우에 의해 처리되었습니다. Jarred는 이 작업에 대해 앞으로 더 자세히 설명할 것입니다.

# **작동 방식**

워크플로우가 시작되면 Claude는 프롬프트에 따라 동적으로 계획을 세우고, 작업을 하위 작업으로 분해한 후, 병렬로 실행되는 하위 에이전트에게 작업을 분산합니다. 결과는 통합되기 전에 검증되며, 사용자는 단일화된 최종 결과물을 받게 됩니다. 에이전트는 독립적인 관점에서 문제를 해결하고, 다른 에이전트는 발견된 내용을 반박하려 시도하며, 결과가 수렴될 때까지 실행이 반복됩니다. 이것이 단일 실행으로는 도달할 수 없는 결과를 워크플로우가 달성하는 방식입니다.

동적 워크플로우는 몇 주가 걸릴 수 있었던 가장 복잡한 엔지니어링 작업을 수행하기 위해 시간과 날짜로 확장될 수 있는 병렬 및 장기 실행 작업을 위해 구축되었습니다. 실행 중에 진행 상황이 저장되므로 중단된 작업은 처음부터 다시 시작하는 대신 중단된 지점부터 계속 진행됩니다. 조정은 대화 외부에서 이루어지기 때문에 작업의 규모가 아무리 커져도 계획은 계속 진행됩니다.

동적 워크플로우는 일반적인 Claude Code 세션보다 의미 있게 더 많은 사용량을 소비한다는 점에 유의하는 것이 중요합니다. 워크플로우가 트리거될 때 Claude Code는 실행될 내용을 보여주고 사용자에게 확인을 요청합니다. 조직 관리자는 관리 설정을 통해 워크플로우를 선택적으로 비활성화할 수도 있습니다.

# **시작하기**

Max 또는 Team 플랜을 사용 중이거나 API를 통해 Claude Code를 사용하는 경우, 동적 워크플로우는 기본적으로 활성화되어 있습니다. Claude에게 워크플로우를 생성하도록 요청하거나 Claude Code 전용 설정인 `ultracode`를 켜서 시작할 수 있습니다. Enterprise 플랜의 경우, 출시 시 동적 워크플로우는 기본적으로 비활성화되어 있습니다. 관리자는 Claude Code [설정](https://code.claude.com/docs/en/settings)에서 이를 쉽게 변경할 수 있습니다.

자세한 내용은 [문서](https://code.claude.com/docs/en/workflows)를 읽어보십시오.
