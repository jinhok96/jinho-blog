> 원문: [Next.js 16.2: AI Improvements](https://nextjs.org/blog/next-16-2-ai) — Next.js Blog

Next.js 16.2는 AI 지원 개발을 위해 설계된 여러 개선 사항을 포함합니다. 이러한 변경 사항을 통해 에이전트는 브라우저 없이도 프로젝트를 더 쉽게 이해하고, 터미널에서 문제를 디버깅하고, 실행 중인 앱을 검사할 수 있습니다.

*   [**Agent-ready `create-next-app`**](#ai-ready-project-setup): 즉시 AI 준비 프로젝트 구성
*   [**Browser Log Forwarding**](#browser-log-forwarding): 에이전트 기반 디버깅을 위해 브라우저 오류를 터미널로 전달
*   [**Dev Server Lock File**](#dev-server-lock-file): 두 번째 개발 서버 시작 시 실행 가능한 오류 메시지
*   [**Experimental Agent DevTools**](#experimental-agent-devtools): AI 에이전트에게 React DevTools 및 Next.js 진단 터미널 액세스 제공

# Agent-ready `create-next-app`[](#agent-ready-create-next-app)

`create-next-app`에는 이제 기본적으로 `AGENTS.md` 파일이 포함되어 프로젝트 시작부터 AI 코딩 에이전트가 버전 일치 Next.js 문서에 액세스할 수 있도록 합니다.

이는 [`AGENTS.md`에 대한 연구](https://vercel.com/blog/agents-md-outperforms-skills-in-our-agent-evals)를 기반으로 하며, 에이전트에게 번들 문서에 액세스할 수 있도록 했을 때 Next.js 평가에서 100% 통과율을 달성했으며, 이는 79%에 달하는 스킬 기반 접근 방식보다 뛰어난 성과를 보였습니다. 핵심 인사이트는 에이전트가 문서를 검색해야 할 시점을 인식하지 못하는 경우가 많기 때문에 항상 사용 가능한 컨텍스트가 온디맨드 검색보다 더 잘 작동한다는 것입니다.

`AGENTS.md` 파일은 에이전트에게 코드를 작성하기 전에 `node_modules/next/dist/docs/`에 번들된 문서를 읽도록 지시하는 짧은 지시문입니다. Next.js npm 패키지에는 이제 전체 문서가 일반 Markdown 파일로 포함되어 에이전트가 외부 데이터를 가져오지 않고도 로컬에서 정확한 버전 일치 참조를 얻을 수 있습니다.

기존 프로젝트의 경우 설정은 Next.js 버전에 따라 달라집니다.

**16.2 이상**에서는 docs가 이미 `next` 패키지에 번들로 제공됩니다. 프로젝트 루트에 다음 두 파일을 추가하세요.

주석 표시는 Next.js 관리 섹션을 구분합니다. 해당 섹션 외부에는 자체 프로젝트별 지침을 추가할 수 있으며, 향후 업데이트에서는 마커 사이의 내용만 대체됩니다.

`CLAUDE.md`는 [Claude Code](https://docs.anthropic.com/en/docs/claude-code)의 지침 파일입니다. `@` 지시문은 추가 컨텍스트로 `AGENTS.md`를 포함하도록 지시합니다.

**이전 버전**의 경우 codemod를 사용하여 이러한 파일을 자동으로 생성하세요.

자세한 내용은 [AI 에이전트 설정 가이드](https://nextjs.org/docs/app/guides/ai-agents#existing-projects)를 참조하세요.

# Browser Log Forwarding[](#browser-log-forwarding)

Next.js는 이제 개발 중에 기본적으로 브라우저 오류를 터미널로 전달하므로 브라우저 콘솔로 전환하지 않고도 클라이언트 측 오류를 볼 수 있습니다. 이는 주로 터미널을 통해 작동하고 브라우저 콘솔에 액세스할 수 없는 AI 에이전트에게 특히 유용합니다.

기본적으로 오류만 터미널로 전달됩니다. `next.config.ts`의 `logging.browserToTerminal`을 사용하여 전달 수준을 제어할 수 있습니다.

# Dev Server Lock File[](#dev-server-lock-file)

Next.js는 이제 실행 중인 개발 서버의 PID, 포트 및 URL을 `.next/dev/lock` 파일에 씁니다. 동일한 프로젝트 디렉토리에서 두 번째 `next dev` 프로세스가 시작되면 Next.js는 잠금 파일을 읽고 실행 가능한 오류를 출력합니다.

이는 서버가 이미 실행 중임을 알지 못한 채 `next dev`를 시작하려고 자주 시도하는 AI 코딩 에이전트에게 특히 유용합니다. 구조화된 오류는 에이전트에게 기존 프로세스를 종료할 PID 또는 연결할 URL을 제공하므로 수동 개입이 필요하지 않습니다.

잠금 파일은 두 개의 `next build` 프로세스가 동시에 실행되는 것을 방지하여 빌드 아티팩트가 손상될 수 있습니다.

# Experimental Agent DevTools[](#experimental-agent-devtools)

위의 기능은 에이전트가 프로젝트를 이해하고 문제를 디버깅하는 데 도움이 됩니다. [`@vercel/next-browser`](https://github.com/vercel-labs/next-browser)는 에이전트가 실행 중인 Next.js 애플리케이션을 검사할 수 있도록 하여 이를 확장합니다.

`next-browser`는 스크린샷, 네트워크 요청, 콘솔 로그와 같은 브라우저 수준 데이터와 React DevTools 및 Next.js 개발 오버레이의 프레임워크별 인사이트(컴포넌트 트리, props, hooks, [Partial Prerendering (PPR)](https://nextjs.org/docs/app/getting-started/cache-components) 셸, 오류 등)를 제공하는 실험적인 CLI입니다. 모두 셸 명령을 통해 구조화된 텍스트로 반환됩니다.

LLM은 DevTools 패널을 읽을 수는 없지만 `next-browser tree`를 실행하고 출력을 구문 분석하여 다음에 검사할 내용을 결정할 수 있습니다. 각 명령은 지속적인 브라우저 세션에 대한 일회성 요청이므로 에이전트는 브라우저 상태를 관리하지 않고도 앱을 반복적으로 쿼리할 수 있습니다. 이는 브라우저를 액세스할 수 없는 UI가 아닌 에이전트가 추론할 수 있는 것으로 만듭니다.

## What it can do today[](#what-it-can-do-today)

기능 세트는 빠르게 발전하고 있습니다. 이 릴리스를 기준으로 `next-browser`는 다음을 지원합니다.

*   **React 컴포넌트 트리 검사** — props, hooks, 상태 및 소스 맵이 지정된 파일 위치 보기
*   **PPR 셸 분석** — 정적 및 동적 영역, 차단된 [Suspense 경계](https://nextjs.org/docs/app/glossary#suspense-boundary) 식별
*   **오류 및 로그 액세스** — 개발 서버에서 빌드 및 런타임 문제 검색
*   **네트워크 활동 모니터링** — 서버 액션을 포함하여 탐색 이후 요청 추적
*   **시각 자료 캡처** — 스크린샷 찍기 또는 로딩 필름스트립 녹화

## Getting started[](#getting-started)

[skill](https://skills.sh/) (AI 에이전트를 위한 재사용 가능한 기능)으로 설치합니다.

그런 다음 Claude Code, Cursor 또는 스킬을 지원하는 모든 AI 에이전트에서 `/next-browser`를 입력합니다. CLI는 React DevTools가 미리 로드된 Chromium 인스턴스를 관리하므로 브라우저 구성이 필요하지 않습니다.

## Example: Growing the static shell[](#example-growing-the-static-shell)

[Partial Prerendering (PPR)](https://nextjs.org/docs/app/getting-started/cache-components)을 사용하면 Next.js는 요청별 데이터에 의존하지 않는 페이지 부분을 에지에서 즉시 [정적 셸](https://nextjs.org/docs/app/glossary#static-shell)로 제공한 다음 나머지를 스트리밍할 수 있습니다. 정적 셸에 더 많은 콘텐츠가 포함될수록 사용자는 더 빨리 의미 있는 페이지를 볼 수 있습니다.

실제로 요청별 가져오기 하나가 실수로 전체 페이지를 동적으로 만들 수 있습니다. 방문자 카운터가 있는 블로그 게시물을 고려해 보세요.

모든 slug는 `generateStaticParams`에 열거되므로 게시물 콘텐츠는 빌드 시 미리 렌더링될 수 있습니다. 그러나 `getVisitorCount`는 각 요청마다 실행되며 컴포넌트의 최상위 수준에 있으므로 _전체 페이지_를 동적으로 만듭니다. 결과적으로 전체 페이지가 점진적으로 스트리밍되는 대신 로딩 스켈레톤 뒤에서 대기합니다.

에이전트는 `next-browser`를 사용하여 이를 진단할 수 있습니다. PPR 모드를 잠그면 정적 셸만 표시됩니다. 이 경우 페이지의 어떤 것도 셸에 포함되지 않았기 때문에 `app/blog/[slug]/loading.tsx` 스켈레톤이 표시됩니다.

그런 다음 에이전트는 `ppr unlock`을 실행하여 무엇이 잘못되었는지 파악합니다.

보고서는 에이전트에게 정확히 무엇을 해야 하는지 알려줍니다. `getVisitorCount`가 차단 요소이며 `BlogPost`에 있으며 수정은 자체 [Suspense 경계](https://nextjs.org/docs/app/glossary#suspense-boundary)로 푸시하는 것입니다. 에이전트는 카운터만 래핑합니다.

`ppr lock`을 다시 실행하면 셸이 확장되었음을 확인합니다. 이제 게시물 콘텐츠가 즉시 미리 렌더링됩니다. 방문자 카운터만 대체 표시됩니다.

`next-browser`는 아직 발전 중이지만, 에이전트가 개발자와 동일한 가시성으로 앱을 디버깅하고 최적화할 수 있는 미래를 가리킵니다.
