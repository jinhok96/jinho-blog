대부분의 AI 앱은 하나의 chat() 호출로 시작됩니다.

하지만 조금 더 복잡한 작업이 필요해지면 이 모든 것이 무너집니다. 하위 에이전트를 도구로 사용하는 방식에 의존하거나, 앱에 동력을 공급할 준수한 워크플로우 또는 오케스트레이션 메커니즘을 만들기 위해 직접 접착제와 추상화를 작성해야 합니다. 이는 실제로 중요하게 생각하는 기능에 집중할 시간을 빼앗을 뿐입니다.

모델은 초안을 작성하고, 비평하고, 수정하고, 승인을 요청하고, 다른 모델을 호출하고, 상태를 업데이트하고, 실행이 진행 중일 때 사용자에게 무슨 일이 일어나고 있는지 보여줘야 합니다. 그 시점에서 일회성 chat 엔드포인트는 fetch 호출, 임시 상태, 사용자 정의 SSE 이벤트 및 아무도 소유하고 싶어하지 않는 많은 코드로 구성된 자체 제작 워크플로우 엔진으로 변합니다.

오늘 TanStack은 실험적인 답변을 소개합니다: **TanStack AI 워크플로우 & 오케스트레이터**.

더 자세히 알아보기 전에, 공정한 경고입니다!

이것은 main에 병합되지 않았습니다. 릴리스되지 않았으며, 안정적이지 않으며, 일반 npm 버전에서는 사용할 수 없습니다. API의 모양이 아직 형성 중인 동안 오늘 pkg.pr.new을 통해 시도해 볼 수 있는 PR 빌드입니다. 목표는 공개 API 모양에 전념하기 전에 실제 사용 사례, 데모 및 피드백에 노출하는 것입니다.

여기서 여러분의 역할이 중요합니다. 여러분의 도움이 필요합니다. 워크플로우와 오케스트레이션 메커니즘을 테스트하고, 여러분의 생각과 의견을 공유하고, 최종 API를 형성하는 데 도움을 주시길 바랍니다.

*   PR: [TanStack/ai#542](https://github.com/TanStack/ai/pull/542)
*   PR 빌드 패키지: [pkg.pr.new 코멘트](https://github.com/TanStack/ai/pull/542#issuecomment-4416347869)
*   패키지: @tanstack/ai-orchestration

목표는 간단합니다: 여러 개의 타입이 지정된 LLM 및 에이전트 단계를 일반 TypeScript async generator로 구성하고, 각 단계를 UI로 스트리밍하고, 인간의 승인을 위해 일시 중지하고, 동일한 SSE 흐름을 통해 다시 시작하는 것입니다.

평가, 데모 및 피드백에만 사용하세요. 공개 API는 안정화 전에 변경될 수 있습니다.

에이전트는 chat() 호출 또는 모든 async 함수를 감싸는 타입이 지정된 래퍼입니다. defineAgent는 각 단계에 입력 스키마, 출력 스키마 및 구현을 제공합니다.

defineAgent는 chat() 호출 또는 입력 및 출력 스키마가 있는 일반 async 함수를 감쌉니다. 워크플로우 런타임은 이러한 스키마를 사용하여 단계에 들어가고 나가는 것을 검증하며, TypeScript는 이를 사용하여 워크플로우 내에서 호출 가능한 모양을 추론합니다. 워크플로우의 관점에서 writer와 editor는 일반적인 타입이 지정된 async 단계입니다.

에이전트가 존재하면 defineWorkflow는 async 함수 내에서 yield\*를 사용하여 이를 구성합니다.

흥미로운 부분은 프레임워크 의례가 없다는 것입니다. TypeScript는 agents.writer에서 기대하는 입력을 알고 있으며, yield\* 후 반환되는 모양도 알고 있습니다. 런타임은 각 yield된 단계 주위에 라이프사이클 이벤트를 내보내고, 실행 중에 텍스트를 스트리밍하고, 결과를 검증하고, 상태를 스냅샷하고, 타입이 지정된 출력으로 generator를 다시 시작할 수 있습니다.

왜 async generator 워크플로우인가? 에이전트 워크플로우를 모델링하는 데는 많은 방법이 있습니다. 그래프 DSL을 구축할 수 있습니다. JSON으로 노드를 정의할 수 있습니다. DAG를 설명하고 런타임이 이를 해석하도록 요청할 수 있습니다. 우리가 generator 워크플로우를 선택한 이유는 에이전트 단계를 yield할 때마다 클라이언트로 직접 스트리밍되기 때문입니다. 사용자는 실시간으로 모든 것을 보고, 최종적으로 최종 출력을 받습니다. 사용자는 도구 호출, 추론 등 발생한 모든 것을 보았습니다.

워크플로우 본문은 그냥 TypeScript입니다. if, for, while, try, await, 헬퍼 함수 및 이미 가지고 있는 도메인 코드를 사용하세요. 오케스트레이션 런타임은 여러분이 yield\*하는 것에만 관심이 있습니다.

각 yield\* agents.someAgent(...)는 타입이 지정된 단계가 됩니다. 런타임은 각 단계 주위에 라이프사이클 이벤트를 내보내고, 실행 중에 텍스트를 스트리밍하고, 결과를 검증하고, 상태를 스냅샷하고, 타입이 지정된 출력으로 generator를 다시 시작할 수 있습니다.

워크플로우는 서버에서 실행됩니다. 브라우저는 이벤트 스트림을 소비합니다.

PR은 @tanstack/ai-orchestration에서 parseWorkflowRequest, runWorkflow, inMemoryRunStore를 추가합니다. 반환된 스트림을 @tanstack/ai의 기존 toServerSentEventsResponse 헬퍼를 통해 파이프할 수 있습니다.

runWorkflow는 실행에 대한 AG-UI 스타일 라이프사이클 이벤트를 내보냅니다. 여기에는 실행 및 단계 이벤트, 상태 스냅샷, JSON Patch 상태 델타, 출력 및 오류가 포함됩니다. UI는 "writer 시작됨", "editor 텍스트 스트리밍됨", "승인 요청됨" 또는 "실행 완료됨"과 같은 이벤트를 위해 자체 이벤트 프로토콜을 발명할 필요가 없습니다.

현재 내장된 지속성은 inMemoryRunStore입니다. 이는 로컬 데모 및 단일 프로세스 평가에 유용합니다. 프로덕션 내구성은 여전히 미래 지향적이며 실험적인 run-store-interface 영역이며, 특히 긴 일시 중지, 배포, 재시작 및 다중 노드 환경의 경우 더욱 그렇습니다. 하지만 API는 자체 내구성이 있는 실행 스토어를 구현하고 준비되었을 때 교체할 수 있도록 제공됩니다.

클라이언트에서는 WorkflowClient, useWorkflow, useOrchestration이 스트리밍된 이벤트를 소비하고 로컬 실행 상태를 최신 상태로 유지합니다.

동일한 런타임과 오케스트레이션 어휘를 위한 useOrchestration도 있습니다. 정신 모델이 "워크플로우 실행"이라면 useWorkflow를 사용하세요. 정신 모델이 "라우터가 다음 에이전트를 선택하도록 허용"이라면 useOrchestration을 사용하세요.

루프 내 인간 제어는 이 PR의 부가 기능이 아닙니다.

워크플로우 코드가 yield\* approve(...)를 호출하면 런타임은 approval-requested 이벤트를 내보내고, 일시 중지된 실행을 실행 스토어에 지속시키고, SSE 응답을 닫습니다. HTTP 요청이 완료됩니다. 서버는 사용자가 생각하거나, diff를 확인하거나, 다른 사람으로부터 승인을 받는 동안 소켓을 열어두지 않습니다.

클라이언트가 approve()를 호출하면 동일한 엔드포인트로 실행 ID와 승인 결과와 함께 POST 요청을 보냅니다. runWorkflow는 generator를 다시 시작하고 다음 SSE 응답은 일시 중지된 지점부터 계속됩니다.

이는 워크플로우 자체 내에서 승인, 수정 및 거부 피드백을 모델링할 수 있음을 의미합니다:

동일한 이벤트 스트림은 상태 스냅샷 및 JSON Patch 델타도 전달하므로 UI는 최종 결과를 기다리지 않고 라이브 상태 검사기, 타임라인 또는 초안 미리 보기를 표시할 수 있습니다.

파이프라인을 알고 있을 때 워크플로우를 사용하세요.

Writer, then editor, then approval은 워크플로우입니다. 토픽 추출, 개요 초안 작성, 섹션 확장도 워크플로우입니다. 정적 검사 실행, 승인 요청, 스테이징 배포, 다시 요청, 프로덕션 배포도 워크플로우입니다.

다음 단계를 턴별로 선택해야 할 때 오케스트레이터를 사용하세요.

defineOrchestrator는 워크플로우를 얇게 감싸므로 위에서 설명한 워크플로우 동작이 오케스트레이터에도 동일하게 적용됩니다. 동일한 런타임 조각을 사용합니다: 타입이 지정된 에이전트, 스트리밍 단계, 상태 스냅샷, 승인, SSE 전송 및 React hook.

본문은 라우터입니다. 라우터는 입력, 변경 가능한 상태 및 이전 결과를 보고 다음 실행할 에이전트를 반환합니다. 라우터가 완료되면 최종 출력을 반환합니다.

라우터는 결정에 비동기 작업이 필요한 경우 yield할 수도 있지만, 반환 값이 중요합니다: 실행할 다음 { agent, input } 쌍 또는 오케스트레이션이 완료되었을 때 { done, output }.

이는 다른 제어 흐름 스타일로 동일한 동작을 제공합니다. 워크플로우는 생성자 본문에 시퀀스를 직접 넣습니다. 오케스트레이터는 다음 단계 결정을 라우터에 넣습니다.

이름 붙일 만큼 중요한 거의 모든 것.

패키지 이름은 @tanstack/ai-orchestration이고, 공개 기능 이름은 TanStack AI Workflows & Orchestrators이며, 핵심 모양은 PR 542에서 볼 수 있습니다. 하지만 이것은 여전히 PR 빌드입니다.

다음과 같은 사항에 대한 면밀한 검토가 예상됩니다:

*   API 이름 및 반환 모양
*   inMemoryRunStore를 넘어서는 내구성 있는 지속성
*   배포 및 다중 서버 노드에 걸친 다시 시작 동작
*   이벤트 이름 및 AG-UI 호환성 세부 사항
*   실행 스토어 인터페이스의 어느 정도를 지금 공개해야 하는가
*   예제가 보증을 과장하지 않고 프로덕션 사용을 어떻게 안내해야 하는가

현재 구현은 시도해 볼 만큼 유용하고 변경될 만큼 이르렀습니다.

다단계 AI 제품 흐름을 구축하고 있다면 지금이 그 모양을 테스트할 때입니다.
