> 원문: [Announcing TypeScript 6.0 RC](https://devblogs.microsoft.com/typescript/announcing-typescript-6-0-rc/) — TypeScript Blog

TypeScript 6.0의 릴리스 후보(RC)를 발표하게 되어 기쁩니다! RC를 시작하려면 다음 명령어를 사용하여 npm을 통해 설치할 수 있습니다.

```
npm install -D typescript@rc

```

TypeScript 6.0은 현재 JavaScript 코드베이스에 기반한 마지막 릴리스라는 점에서 독특합니다. 작년에 발표된 바와 같이, 네이티브 코드의 속도와 공유 메모리 멀티스레딩을 활용하는 Go로 작성된 TypeScript 컴파일러 및 언어 서비스의 새로운 코드베이스를 작업 중입니다. 이 새로운 코드베이스는 TypeScript 7.0 및 그 이후의 기반이 될 것입니다. TypeScript 6.0은 해당 릴리스의 즉각적인 전신이 될 것이며, 많은 면에서 TypeScript 5.9와 7.0 사이의 다리 역할을 할 것입니다. 따라서 TypeScript 6.0의 대부분의 변경 사항은 TypeScript 7.0 채택을 조정하고 준비하는 데 중점을 두고 있습니다.

그럼에도 불구하고, 정렬뿐만 아니라 몇 가지 새로운 기능과 개선 사항이 있습니다. 이 릴리스의 주요 내용을 살펴본 후, 7.0에 변경될 사항과 준비 방법에 대해 더 자세히 살펴보겠습니다.

# 베타 이후 새로운 기능은 무엇인가요?

TypeScript 6.0 베타 이후 몇 가지 주목할 만한 변경 사항이 있었습니다. 대부분 TypeScript 7.0의 동작과 일치시키기 위한 것입니다.

한 가지 조정 사항은 제네릭 호출에서의 함수 표현식의 타입 검사입니다. 특히 제네릭 JSX 표현식에서 발생하는 경우에 해당합니다. 이는 일반적으로 기존 코드에서 더 많은 버그를 포착할 것이지만, 일부 제네릭 호출에는 명시적 타입 인수가 필요할 수 있습니다.

또한 `import()` 호출에 대한 import assertion 구문(`import ... assert {...}`)의 사용 중단을 확장했습니다.

마지막으로, 최신 웹 표준을 반영하여 DOM 타입을 업데이트했으며, Temporal API에 대한 몇 가지 조정도 포함되었습니다.

# `this`-없는 함수의 컨텍스트 민감도 감소

매개변수에 명시적 타입이 작성되지 않은 경우, TypeScript는 예상되는 타입에 기반하거나 동일한 함수 호출의 다른 인수를 통해 추론할 수 있습니다.

```
declare function callIt<T>(obj: {
    produce: (x: number) => T,
    consume: (y: T) => void,
}): void;

// 문제 없이 작동합니다.
callIt({
    produce: (x: number) => x * 2,
    consume: y => y.toFixed(),
});

// 속성의 순서가 바뀌어도 문제 없이 작동합니다.
callIt({
    consume: y => y.toFixed(),
    produce: (x: number) => x * 2,
});

```

여기서 TypeScript는 속성의 순서에 관계없이 `produce` 함수에서 추론된 `T`를 기반으로 `consume` 함수의 `y` 타입을 추론할 수 있습니다. 그러나 이러한 함수가 화살표 함수 구문 대신 _메서드 구문_을 사용하여 작성되었다면 어떻게 될까요?

```
declare function callIt<T>(obj: {
    produce: (x: number) => T,
    consume: (y: T) => void,
}): void;

// 정상적으로 작동하며, `x`는 number로 추론됩니다.
callIt({
    produce(x: number) { return x * 2; },
    consume(y) { return y.toFixed(); },
});

callIt({
    consume(y) { return y.toFixed(); },
    //                  ~
    // 오류: 'y'는 'unknown' 타입입니다.

    produce(x: number) { return x * 2; },
});

```

이상하게도 `callIt`에 대한 두 번째 호출은 TypeScript가 `consume` 메서드의 `y` 타입을 추론할 수 없기 때문에 오류를 발생시킵니다. 여기서 발생하는 문제는 TypeScript가 `T`에 대한 후보를 찾으려고 할 때, 매개변수에 명시적 타입이 없는 함수는 먼저 건너뛴다는 것입니다. 이는 특정 함수가 올바르게 검사되기 위해 `T`의 추론된 타입이 필요할 수 있기 때문입니다. 이 경우 `consume` 함수를 분석하기 위해 `T`의 타입을 알아야 합니다.

이러한 함수를 _컨텍스트 민감 함수_라고 합니다. 기본적으로 매개변수에 명시적 타입이 없는 함수입니다. 궁극적으로 타입 시스템은 이러한 매개변수에 대한 타입을 파악해야 하지만, 이는 타입 추론 방식과 약간 충돌합니다. 제네릭 함수의 경우, 두 가지가 다른 방향으로 타입을 "당기기" 때문입니다.

```
function callFunc<T>(callback: (x: T) => void, value: T) {
    return callback(value);
}

callFunc(x => x.toFixed(), 42);
//       ^
// 여기서 `x`의 타입을 파악해야 하지만,
// 콜백을 검사하기 위해 `T`의 타입을 파악해야 합니다.

```

이를 해결하기 위해 TypeScript는 타입 인수 추론 중에 컨텍스트 민감 함수를 건너뛰고 대신 다른 인수를 먼저 확인하고 추론합니다. 컨텍스트 민감 함수를 건너뛰는 것이 작동하지 않으면, 추론은 단순히 검사되지 않은 인수 전체를 왼쪽에서 오른쪽으로 진행합니다. 바로 위 예에서 TypeScript는 `T` 추론 중에 콜백을 건너뛰지만, 두 번째 인수 `42`를 보고 `T`가 `number`라고 추론합니다. 그런 다음 콜백 검사로 돌아가면 `(x: number) => void`의 컨텍스트 타입이 있어 `x`도 `number`라고 추론할 수 있게 됩니다.

그렇다면 이전 예에서 무슨 일이 일어나고 있는 걸까요?

```
// 화살표 구문 - 오류 없음.
callIt({
    consume: y => y.toFixed(),
    produce: (x: number) => x * 2,
});

// 메서드 구문 - 오류 발생!
callIt({
    consume(y) { return y.toFixed(); },
    //                  ~
    // 오류: 'y'는 'unknown' 타입입니다.

    produce(x: number) { return x * 2; },
});

```

두 예 모두 `produce`에는 명시적으로 타입이 지정된 `x` 매개변수가 있는 함수가 할당되어 있습니다. 동일하게 검사되어야 하는 것이 아닙니까?

이 문제는 미묘합니다. 대부분의 함수(메서드 구문을 사용하는 함수 포함)에는 암시적인 `this` 매개변수가 있지만, 화살표 함수에는 없습니다. `this`의 사용은 `T`의 타입에 "당기는" 것을 요구할 수 있습니다. 예를 들어, 포함된 객체 리터럴의 타입을 아는 것은 차례로 `T`를 사용하는 `consume`의 타입이 필요할 수 있습니다.

하지만 `this`를 사용하지 않고 있습니다! 물론 런타임에 함수에 `this` 값이 있을 수 있지만, 사용되지 않습니다!

TypeScript 6.0은 함수가 컨텍스트 민감 함수인지 아닌지를 결정할 때 이를 고려합니다. `this`가 함수에서 실제로 _사용되지 않는_ 경우, 컨텍스트 민감 함수로 간주되지 않습니다. 이는 이러한 함수가 타입 추론에서 더 높은 우선순위를 갖게 된다는 것을 의미하며, 위의 모든 예가 이제 작동합니다!

[이 변경 사항은 Mateusz Burzyński의 작업 덕분에 제공되었습니다.](https://github.com/microsoft/TypeScript/pull/62243)

# `#/`로 시작하는 서브패스 임포트

Node.js가 모듈 지원을 추가했을 때, ["서브패스 임포트"](https://nodejs.org/api/packages.html#subpath-imports)라는 기능을 추가했습니다. 이것은 기본적으로 ["imports"](https://nodejs.org/api/packages.html#imports)라는 필드로, 패키지가 패키지 내의 모듈에 대한 내부 별칭을 만들 수 있도록 합니다.

```
{
    "name": "my-package",
    "type": "module",
    "imports": {
        "#root": "./dist/index.js",
        "#root/*": "./dist/*"
    }
}

```

이를 통해 `my-package`의 모듈은 `../../index.js`와 같은 상대 경로를 사용할 필요 없이 `#root`에서 임포트할 수 있으며, 기본적으로 다른 모듈이 다음과 같이 작성할 수 있습니다.

```
import * as utils from "#root/utils.js";

```

다음과 같은 상대 경로를 사용하는 대신.

```
import * as utils from "../../utils.js";

```

이 기능의 사소한 불편함 중 하나는 개발자가 서브패스 임포트를 지정할 때 항상 `#` 뒤에 _무언가_를 작성해야 한다는 것이었습니다. 여기서는 `root`를 사용했지만, `./dist/` 외에 매핑하는 디렉토리가 없기 때문에 약간 쓸모가 없습니다.

번들러를 사용한 개발자들도 긴 상대 경로를 피하기 위해 경로 매핑에 익숙합니다. 번들러의 일반적인 관례는 간단한 `@/`를 접두사로 사용하는 것이었습니다. 안타깝게도 서브패스 임포트는 `#/`로 시작할 수 없었기 때문에 프로젝트에서 이를 채택하려는 개발자들에게 많은 혼란을 야기했습니다.

하지만 최근 Node.js가 `#/`로 시작하는 서브패스 임포트에 대한 지원을 추가했습니다. 이를 통해 패키지는 추가 세그먼트를 추가할 필요 없이 서브패스 임포트에 간단한 `#/` 접두사를 사용할 수 있습니다.

```
{
    "name": "my-package",
    "type": "module",
    "imports": {
        "#": "./dist/index.js",
        "#/*": "./dist/*"
    }
}

```

이는 최신 Node.js 20 릴리스에서 지원되며, TypeScript는 이제 `--moduleResolution` 설정의 `node20`, `nodenext`, `bundler` 옵션에서 이를 지원합니다.

이 작업은 [magic-akari](https://github.com/magic-akari) 덕분에 이루어졌으며, [구현 풀 요청은 여기서 찾을 수 있습니다.](https://github.com/microsoft/TypeScript/pull/62844)

# `--moduleResolution bundler`와 `--module commonjs` 결합

TypeScript의 `--moduleResolution bundler` 설정은 이전에 `--module esnext` 또는 `--module preserve`와 함께만 사용할 수 있었습니다. 그러나 `--moduleResolution node`(a.k.a. `--moduleResolution node10`)의 사용 중단으로 인해, 이 새로운 조합은 많은 프로젝트에 가장 적합한 업그레이드 경로가 되는 경우가 많습니다.

프로젝트는 종종 다음 중 하나로 마이그레이션을 계획하는 것을 선호할 것입니다.

*   `--module preserve` 및 `--moduleResolution bundler`
*   `--module nodenext`

프로젝트 유형(예: 번들된 웹 앱, Bun 앱 또는 Node.js 앱)에 따라 다릅니다.

더 많은 정보는 [이 구현 풀 요청](https://github.com/microsoft/TypeScript/pull/62320)에서 찾을 수 있습니다.

# `--stableTypeOrdering` 플래그

[TypeScript의 네이티브 포트](https://devblogs.microsoft.com/typescript/typescript-native-port/)에 대한 지속적인 작업의 일환으로, 6.0에서 7.0 마이그레이션을 지원하기 위한 새로운 플래그인 `--stableTypeOrdering`을 도입했습니다.

오늘날 TypeScript는 타입에 따라 타입 ID(내부 추적 번호)를 할당하고 이러한 ID를 사용하여 유니온 타입을 일관되게 정렬합니다. 유사한 프로세스가 속성에 대해 발생합니다. 결과적으로 프로그램에서 항목이 선언되는 순서는 선언 내보내기와 같은 것에 놀라운 영향을 미칠 수 있습니다.

예를 들어, 이 파일의 선언 내보내기를 고려해 보겠습니다.

```
// 입력: some-file.ts
export function foo(condition: boolean) {
    return condition ? 100 : 500;
}

// 출력: some-file.d.ts
export declare function foo(condition: boolean): 100 | 500;
//                                               ^^^^^^^^^
//             참고: 이 유니온의 순서: 100, 다음 500.

```

`foo` 위에 관련 없는 `const`를 추가하면 선언 내보내기가 변경됩니다.

```
// 입력: some-file.ts
const x = 500;
export function foo(condition: boolean) {
    return condition ? 100 : 500;
}

// 출력: some-file.d.ts
export declare function foo(condition: boolean): 500 | 100;
//                                               ^^^^^^^^^
//                           참고: 여기서 순서 변경.

```

이는 리터럴 타입 `500`이 `const x` 선언을 분석할 때 먼저 처리되었기 때문에 `100`보다 낮은 타입 ID를 받기 때문입니다. 매우 드문 경우에 이 순서 변경은 프로그램 처리 순서에 따라 오류가 나타나거나 사라질 수 있지만, 일반적으로 이 순서를 가장 눈에 띄게 알 수 있는 곳은 내보낸 선언 파일이거나 편집기에서 타입이 표시되는 방식입니다.

TypeScript 7의 주요 아키텍처 개선 사항 중 하나는 병렬 타입 검사로, 전반적인 검사 시간을 크게 향상시킵니다. 그러나 병렬성은 문제를 야기합니다. 서로 다른 타입 검사기가 노드, 타입 및 심볼을 다른 순서로 방문할 때, 이러한 구문에 할당된 내부 ID는 결정론적이지 않게 됩니다. 이는 혼란스러운 비결정적 출력으로 이어지며, 동일한 프로그램에서 동일한 내용을 가진 두 파일이 다른 선언 파일을 생성하거나 동일한 파일을 분석할 때 다른 오류를 계산할 수 있습니다. 이를 해결하기 위해 TypeScript 7.0은 객체의 내용에 기반한 결정론적 알고리즘에 따라 내부 객체(예: 타입 및 심볼)를 정렬합니다. 이를 통해 모든 검사기가 생성 시점과 관계없이 동일한 객체 순서를 접하게 됩니다. 결과적으로, 주어진 예에서 TypeScript는 _항상_ `100 | 500`을 출력하여 순서 불안정성을 완전히 제거합니다.

이는 TypeScript 6과 7이 때때로 다른 순서를 표시할 수 있음을 의미합니다. 이러한 순서 변경은 거의 항상 무해하지만, 컴파일러 출력 간의 비교(예: 6.0 대 7.0의 내보낸 선언 파일 확인)를 하는 경우, 이러한 다른 순서는 정확성을 평가하기 어렵게 만드는 많은 노이즈를 생성할 수 있습니다. 그러나 가끔은 타입 오류가 나타나거나 사라지는 순서 변경을 목격할 수 있으며, 이는 더욱 혼란스러울 수 있습니다.

이 상황을 돕기 위해 6.0에서는 새로운 `--stableTypeOrdering` 플래그를 지정할 수 있습니다. 이는 6.0의 타입 순서 동작을 7.0의 동작과 일치시켜 두 코드베이스 간의 차이점을 줄입니다. 이 플래그를 항상 사용하는 것이 반드시 권장되지는 않습니다. 타입 검사에 상당한 속도 저하를 유발할 수 있기 때문입니다(코드베이스에 따라 최대 25%).

`--stableTypeOrdering`을 사용하여 타입 오류가 발생하는 경우, 이는 일반적으로 추론 차이 때문입니다. `--stableTypeOrdering`이 없는 이전 추론은 프로그램의 현재 타입 순서에 따라 작동했습니다. 이를 돕기 위해 명시적 타입을 어디에든 제공하는 것이 도움이 될 것입니다. 종종 이는 타입 인수일 것입니다.

```
- someFunctionCall(/*...*/);
+ someFunctionCall<SomeExplicitType>(/*...*/);

```

또는 호출에 전달하려는 인수에 대한 변수 주석입니다.

```
- const someVariable = { /*... some complex object ...*/ };
+ const someVariable: SomeExplicitType = { /*... some complex object ...*/ };

someFunctionCall(someVariable);

```

**이 플래그는 6.0과 7.0 간의 차이점을 진단하는 데만 사용되며, 장기 기능으로 사용하기 위한 것이 아님을 명심하십시오.**

[이 풀 요청에서 더 자세히 알아보세요.](https://github.com/microsoft/TypeScript/pull/63084)

# `target` 및 `lib`에 대한 `es2025` 옵션

TypeScript 6.0은 `target` 및 `lib` 모두에 대해 `es2025` 옵션을 지원합니다. ES2025에는 새로운 JavaScript 언어 기능이 없지만, 이 새로운 타겟은 내장 API(예: `RegExp.escape`)에 대한 새로운 타입을 추가하고 몇 가지 선언을 `esnext`에서 `es2025`로 이동합니다(예: `Promise.try`, `Iterator` 메서드 및 `Set` 메서드). [새로운 타겟을 활성화하기 위한 작업](https://github.com/microsoft/TypeScript/pull/63046)은 [Kenta Moriuchi](https://github.com/petamoriken)의 기여로 이루어졌습니다.

# `Temporal`용 새 타입

오랫동안 기다려온 [Temporal 제안](https://github.com/tc39/proposal-temporal)이 스테이지 3에 도달했으며 곧 JavaScript에 추가될 예정입니다. TypeScript 6.0은 Temporal API에 대한 내장 타입을 포함하므로 `--target esnext` 또는 `"lib": ["esnext"]`(또는 더 세분화된 `esnext.temporal`)을 통해 오늘날 TypeScript 코드에서 이를 사용하기 시작할 수 있습니다.

```
let yesterday = Temporal.Now.instant().subtract({
    hours: 24,
});

let tomorrow = Temporal.Now.instant().add({
    hours: 24,
});

console.log(`Yesterday: ${yesterday}`);
console.log(`Tomorrow: ${tomorrow}`);

```

Temporal은 이미 여러 런타임에서 사용할 수 있으므로 곧 실험을 시작할 수 있을 것입니다. [Temporal API에 대한 문서는 MDN에서 확인할 수 있습니다.](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Temporal) 하지만 아직 불완전할 수 있습니다.

[이 작업은 GitHub 사용자 Renegade334의 기여로 이루어졌습니다.](https://github.com/microsoft/TypeScript/pull/62628)

# "upsert" 메서드(a.k.a. `getOrInsert`)용 새 타입

`Map`과 관련된 일반적인 패턴은 키가 있는지 확인하고, 없으면 기본값을 설정하고 가져오는 것입니다.

```
function processOptions(compilerOptions: Map<string, unknown>) {
    let strictValue: unknown;
    if (compilerOptions.has("strict")) {
        strictValue = compilerOptions.get("strict");
    }
    else {
        strictValue = true;
        compilerOptions.set("strict", strictValue);
    }
    // ...
}

```

이 패턴은 지루할 수 있습니다. [ECMAScript의 "upsert" 제안](https://github.com/tc39/proposal-upsert)이 최근 스테이지 4에 도달했으며, `Map` 및 `WeakMap`에 두 가지 새로운 메서드를 도입했습니다.

*   `getOrInsert`
*   `getOrInsertComputed`

이러한 메서드는 `esnext` 라이브러리에 추가되어 TypeScript 6.0에서 즉시 사용할 수 있습니다.

`getOrInsert`를 사용하면 위의 코드를 다음과 같이 바꿀 수 있습니다.

```
function processOptions(compilerOptions: Map<string, unknown>) {
    let strictValue = compilerOptions.getOrInsert("strict", true);
    // ...
}

```

`getOrInsertComputed`는 유사하게 작동하지만, 기본값이 계산하기에 비용이 많이 드는 경우(예: 많은 계산, 할당이 필요하거나 오래 실행되는 동기 I/O 수행)에 사용됩니다. 대신, 키가 아직 존재하지 않는 경우에만 호출되는 콜백을 받습니다.

```
someMap.getOrInsertComputed("someKey", () => {
    return computeSomeExpensiveValue(/*...*/);
});

```

이 콜백은 키를 인수로 제공하므로 기본값이 키에 기반하는 경우 유용할 수 있습니다.

```
someMap.getOrInsertComputed(someKey, computeSomeExpensiveDefaultValue);

function computeSomeExpensiveValue(key: string) {
    // ...
}

```

[이 업데이트는 GitHub 사용자 Renegade334의 기여로 이루어졌습니다.](https://github.com/microsoft/TypeScript/pull/62612)

# `RegExp.escape`

정규 표현식 내에서 일치시킬 리터럴 문자열을 구성할 때 `*`, `+`, `?`, `(`, `)` 등과 같은 특수 정규 표현식 문자를 이스케이프하는 것이 중요합니다. [RegExp Escaping ECMAScript 제안](https://github.com/tc39/proposal-regex-escaping)이 스테이지 4에 도달했으며, 이를 처리하는 새로운 `RegExp.escape` 함수를 도입합니다.

```
function matchWholeWord(word: string, text: string) {
    const escapedWord = RegExp.escape(word);
    const regex = new RegExp(`\\b${escapedWord}\\b`, "g");
    return text.match(regex);
}

```

`RegExp.escape`는 `es2025` 라이브러리에서 사용할 수 있으므로 오늘부터 TypeScript 6.0에서 사용할 수 있습니다.

[이 작업은 Kenta Moriuchi의 기여로 이루어졌습니다.](https://github.com/microsoft/TypeScript/pull/63046)

# `dom` 라이브러리에 `dom.iterable` 및 `dom.asynciterable` 포함

TypeScript의 `lib` 옵션은 대상 런타임에 어떤 전역 선언이 있는지 지정할 수 있도록 합니다. 한 가지 옵션은 웹 환경(예: DOM API를 구현하는 브라우저)을 나타내는 `dom`입니다. 이전에 DOM API는 `Iterable` 및 `AsyncIterable`을 지원하지 않는 환경에 대해 `dom.iterable` 및 `dom.asynciterable`로 부분적으로 분리되었습니다. 이는 `NodeList` 또는 `HTMLCollection`과 같은 DOM 컬렉션에서 반복 메서드를 사용하기 위해 명시적으로 `dom.iterable`을 추가해야 함을 의미했습니다.

TypeScript 6.0에서는 `lib.dom.iterable.d.ts` 및 `lib.dom.asynciterable.d.ts`의 내용이 `lib.dom.d.ts`에 완전히 포함됩니다. 구성 파일의 `"lib"` 배열에서 `dom.iterable` 및 `dom.asynciterable`을 계속 참조할 수 있지만, 이제는 비어 있는 파일입니다.

```
// TypeScript 6.0 이전에는 "lib": ["dom", "dom.iterable"]이 필요했습니다.
// 이제 "lib": ["dom"]만으로 작동합니다.
for (const element of document.querySelectorAll("div")) {
    console.log(element.textContent);
}

```

이는 일반적인 혼란 지점을 제거하는 삶의 질 개선 사항입니다. 어떤 최신 브라우저도 이러한 기능을 지원하지 않는 경우는 거의 없기 때문입니다. 이미 `dom`과 `dom.iterable`을 모두 포함하고 있었다면 이제 `dom`만으로 단순화할 수 있습니다.

[이 이슈](https://github.com/microsoft/TypeScript/issues/60959)와 [해당 풀 요청](https://github.com/microsoft/TypeScript/pull/62111)에서 더 자세히 알아보세요.

# TypeScript 6.0의 호환성 변경 및 사용 중단

TypeScript 6.0은 TypeScript 컴파일러의 네이티브 포트인 TypeScript 7.0을 준비하도록 설계된 중요한 전환 릴리스로 제공됩니다. TypeScript 6.0은 기존 TypeScript 지식과의 완전한 호환성을 유지하고 TypeScript 5.9와 API 호환성을 계속 유지하지만, 이 릴리스는 진화하는 JavaScript 생태계를 반영하고 TypeScript 7.0의 기반을 마련하는 여러 호환성 변경 및 사용 중단을 도입합니다.

TypeScript 5.0 이후 2년 동안 JavaScript를 작성하고 배포하는 방식에 지속적인 변화가 있었습니다.

*   사실상 모든 런타임 환경이 "evergreen"이 되었습니다. 진정한 레거시 환경(ES5)은 극히 드물게 되었습니다.
*   번들러와 ESM은 새로운 프로젝트의 가장 일반적인 모듈 타겟이 되었지만, CommonJS는 여전히 주요 타겟입니다. AMD 및 기타 브라우저 내 사용자 영역 모듈 시스템은 2012년보다 훨씬 드물어졌습니다.
*   거의 모든 패키지를 일부 모듈 시스템을 통해 소비할 수 있습니다. UMD 패키지는 여전히 존재하지만, _유일하게_ 전역 변수로 제공되는 새 코드는 거의 없습니다.
*   `tsconfig.json`은 거의 보편적인 구성 메커니즘입니다.
*   "더 엄격한" 타입에 대한 수요는 계속 증가하고 있습니다.
*   TypeScript 빌드 성능이 가장 중요합니다. TypeScript 7의 이점에도 불구하고 성능은 항상 주요 목표로 유지되어야 하며, 성능을 지원할 수 없는 옵션은 더 강력하게 정당화되어야 합니다.

따라서 TypeScript 6.0과 7.0은 이러한 현실을 염두에 두고 설계되었습니다. TypeScript 6.0의 경우, 이러한 사용 중단은 `tsconfig`에서 `"ignoreDeprecations": "6.0"`을 설정하여 무시할 수 있습니다. 그러나 TypeScript 7.0은 이러한 사용 중단된 옵션을 _지원하지 않을_ 것임을 유의하십시오.

일부 필요한 조정은 코드 변환기 또는 도구를 사용하여 자동으로 수행할 수 있습니다. 예를 들어, [실험적인 `ts5to6` 도구](https://github.com/andrewbranch/ts5to6)는 코드베이스 전체에서 `baseUrl` 및 `rootDir`를 자동으로 조정할 수 있습니다.

## 선결제 조정

아래에서 특정 조정을 다룰 것이지만, 일부 사용 중단 및 동작 변경에는 관련 문제를 직접 가리키는 오류 메시지가 반드시 있는 것은 아니라는 점을 명심해야 합니다. 따라서 **대부분의 프로젝트는 다음 중 적어도 하나를 수행해야 합니다.**

*   `tsconfig`에서 `"types"` 배열을 설정합니다. 일반적으로 `"types": ["node"]`입니다.
    
    `"types": ["*"]`는 5.9 동작을 복원하지만, 빌드 성능 및 예측 가능성을 개선하기 위해 명시적 배열을 사용하는 것이 좋습니다.
    
    일반적으로 식별자가 누락되거나 확인되지 않은 내장 모듈과 관련된 _많은_ 타입 오류가 발생하는 경우 이 문제가 발생했음을 알 수 있습니다.
    
*   이전에 추론에 의존했다면 `"rootDir": "./src"`를 설정합니다.
    
    파일이 `./dist/index.js` 대신 `./dist/src/index.js`에 기록되는 것을 보는 경우 이 문제가 발생하는 경우가 많습니다.
    

## 간단한 기본값 변경

몇 가지 컴파일러 옵션의 기본값이 업데이트되어 현대 개발 관행을 더 잘 반영합니다.

*   **`strict`가 이제 기본적으로 `true`입니다**: 더 엄격한 타입에 대한 수요가 계속 증가하고 있으며, 대부분의 새 프로젝트에서 `strict` 모드를 활성화하기를 원한다는 것을 발견했습니다. 이미 `"strict": true`를 사용하고 있었다면 아무것도 변경되지 않습니다. 이전 기본값인 `false`에 의존했다면 `tsconfig.json`에 명시적으로 `"strict": false`를 설정해야 합니다.
    
*   **`module`이 `esnext`로 기본 설정됩니다**: 마찬가지로, 새 기본값인 `module`은 `esnext`이며, ESM이 이제 지배적인 모듈 형식임을 인정합니다.
    
*   **`target`이 현재 연도의 ES 버전으로 기본 설정됩니다**: 새 기본값인 `target`은 지원되는 최신 ECMAScript 사양 버전입니다(기본적으로 부동 타겟). 현재 해당 타겟은 `es2025`입니다. 이는 대부분의 개발자가 evergreen 런타임으로 배포하고 있으며 이전 ECMAScript 버전으로 트랜스파일할 필요가 없다는 현실을 반영합니다.
    
*   **`noUncheckedSideEffectImports`가 이제 기본적으로 `true`입니다**: 이는 부작용 전용 임포트의 오타 문제를 포착하는 데 도움이 됩니다.
    
*   **`libReplacement`이 이제 기본적으로 `false`입니다**: 이 플래그는 이전에는 모든 실행에 대해 많은 실패한 모듈 확인을 발생시켰으며, 이는 `--watch` 및 편집기 시나리오에서 감시해야 하는 위치 수를 증가시켰습니다. 새 프로젝트에서 `libReplacement`는 다른 명시적 구성이 수행될 때까지 아무 작업도 수행하지 않으므로 기본적으로 성능 향상을 위해 기본적으로 비활성화하는 것이 합리적입니다.
    

이러한 새 기본값이 프로젝트를 중단시키는 경우, `tsconfig.json`에서 이전 값을 명시적으로 지정할 수 있습니다.

## `rootDir`이 이제 `.`으로 기본 설정됩니다

`rootDir`은 출력 디렉토리에 상대적인 출력 파일의 디렉토리 구조를 제어합니다. 이전에는 `rootDir`을 지정하지 않으면 선언 파일이 아닌 모든 입력 파일의 공통 디렉토리를 기반으로 추론되었습니다. 그러나 이는 종종 파일이 프로젝트에 속하는지 여부를 해당 프로젝트를 로드하고 구문 분석하기 전에는 알 수 없음을 의미했습니다. 또한 TypeScript가 프로그램의 모든 파일 경로를 분석하여 해당 공통 소스 디렉토리를 추론하는 데 더 많은 시간을 소비해야 함을 의미했습니다.

TypeScript 6.0에서는 기본 `rootDir`이 항상 `tsconfig.json` 파일이 포함된 디렉토리가 됩니다. `rootDir`은 `tsconfig.json` 파일 없이 명령줄에서 `tsc`를 사용할 때만 추론됩니다.

`tsconfig.json` 디렉토리보다 더 깊은 수준에 소스 파일이 있고 소스 파일의 공통 루트 디렉토리를 추론하는 데 TypeScript에 의존했다면 `rootDir`을 명시적으로 설정해야 합니다.

```
  {
      "compilerOptions": {
          // ...
+         "rootDir": "./src"
      },
      "include": ["./src"]
  }

```

마찬가지로 `tsconfig.json`이 포함된 `tsconfig.json` 외부 파일을 참조하는 경우 해당 파일을 포함하도록 `rootDir`을 조정해야 합니다.

```
  {
      "compilerOptions": {
          // ...
+         "rootDir": "../src"
      },
      "include": ["../src/**/*.tests.ts"]
  }

```

[여기에서 토론](https://github.com/microsoft/TypeScript/issues/62194) 및 [여기에서 구현](https://github.com/microsoft/TypeScript/pull/62418)에서 더 자세히 알아보세요.

## `types`가 이제 `[]`으로 기본 설정됩니다

`tsconfig.json`에서 `compilerOptions`의 `types` 필드는 컴파일 중에 전역 범위에 포함될 패키지 이름 목록을 지정합니다. 일반적으로 `node_modules`의 패키지는 소스 코드의 임포트를 통해 자동으로 포함됩니다. 그러나 편의를 위해 TypeScript는 기본적으로 `@types`의 모든 패키지를 포함했으므로, `@types/node`의 `process` 또는 `"fs"` 모듈과 같은 전역 선언이나 `@types/jest`의 `describe` 및 `it`을 직접 임포트하지 않고도 가져올 수 있습니다.

어떤 의미에서 `types` 값은 이전에 "`@types`의 모든 것을 열거"하는 것으로 기본 설정되었습니다. 이는 일반적인 리포지토리 설정이 오늘날 수백 개의 `@types` 패키지를 연결할 수 있고, 특히 여러 프로젝트 워크스페이스에서 `node_modules`가 평탄화된 경우 매우 비쌀 수 있습니다. 현대 프로젝트는 거의 항상 `@types/node`, `@types/jest` 또는 기타 몇 가지 일반적인 전역 영향을 미치는 패키지만 필요합니다.

TypeScript 6.0에서는 기본 `types` 값이 `[]`(빈 배열)이 됩니다. 이 변경으로 인해 프로젝트가 빌드 시점에 의도치 않게 수백 또는 수천 개의 불필요한 선언 파일을 가져오는 것을 방지합니다. 우리가 검토한 많은 프로젝트는 `types`를 적절하게 설정하기만 해도 빌드 시간이 20-50% 개선되었습니다.

**이는 많은 프로젝트에 영향을 미칩니다.** `"types": ["node"]` 또는 다른 몇 가지를 추가해야 할 가능성이 높습니다.

```
  {
      "compilerOptions": {
          // 필요한 @types 패키지를 명시적으로 나열합니다.
+         "types": ["node", "jest"]
      }
  }

```

`*` 항목을 지정하여 이전 열거 동작을 다시 활성화할 수도 있습니다.

```
  {
      "compilerOptions": {
          // 모든 타입 로드 - TypeScript 5.9 이전의 기본값입니다.
+         "types": ["*"]
      }
  }

```

다음과 같은 새 오류 메시지가 발생하는 경우:

```
'...' 모듈 또는 해당 타입 선언을 찾을 수 없습니다.
'fs' 이름을 찾을 수 없습니다. node에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/node`를 실행한 다음 tsconfig의 types 필드에 'node'를 추가해 보세요.
'path' 이름을 찾을 수 없습니다. node에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/node`를 실행한 다음 tsconfig의 types 필드에 'node'를 추가해 보세요.
'process' 이름을 찾을 수 없습니다. node에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/node`를 실행한 다음 tsconfig의 types 필드에 'node'를 추가해 보세요.
'Bun' 이름을 찾을 수 없습니다. Bun에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/bun`을 실행한 다음 tsconfig의 types 필드에 'bun'을 추가해 보세요.
'describe' 이름을 찾을 수 없습니다. 테스트 실행기에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/jest` 또는 `npm i --save-dev @types/mocha`를 실행한 다음 tsconfig의 types 필드에 'jest' 또는 'mocha'를 추가해 보세요.

```

`types` 필드에 몇 가지 항목을 추가해야 할 가능성이 높습니다.

[여기에서 제안](https://github.com/microsoft/TypeScript/issues/62195)과 [구현 풀 요청](https://github.com/microsoft/TypeScript/pull/63054)에서 더 자세히 알아보세요.

## 사용 중단: `target: es5`

ECMAScript 5 타겟은 오랫동안 레거시 브라우저를 지원하는 데 중요했지만, 그 후속인 ECMAScript 2015(ES6)은 10년 이상 전에 출시되었으며, 모든 최신 브라우저는 수년 동안 이를 지원해 왔습니다. Internet Explorer의 은퇴와 evergreen 브라우저의 보편화로 인해 오늘날 ES5 출력에 대한 사용 사례는 거의 없습니다.

TypeScript의 가장 낮은 타겟은 이제 ES2015가 되며, `target: es5` 옵션은 사용 중단되었습니다. `target: es5`를 사용했다면, 더 최신 타겟으로 마이그레이션하거나 외부 컴파일러를 사용해야 합니다. 여전히 ES5 출력이 필요한 경우, 외부 컴파일러를 사용하여 TypeScript 소스를 직접 컴파일하거나 TypeScript 출력을 후처리하는 것이 좋습니다.

[이 사용 중단에 대한 자세한 내용은 여기서 확인하세요.](https://github.com/microsoft/TypeScript/issues/62196) [해당 구현 풀 요청](https://github.com/microsoft/TypeScript/pull/63067)과 함께.

## 사용 중단: `--downlevelIteration`

`--downlevelIteration`은 ES5 출력에만 영향을 미치며, `--target es5`가 사용 중단되었기 때문에 `--downlevelIteration`은 더 이상 목적이 없습니다.

미묘하게도 TypeScript 5.9 및 이전 버전에서는 `--target es2015`와 함께 `--downlevelIteration false`를 사용하는 것이 오류를 발생시키지 않았습니다. TypeScript 6.0에서는 `--downlevelIteration`을 설정하면 사용 중단 오류가 발생합니다.

[여기에서 구현을 확인하세요.](https://github.com/microsoft/TypeScript/pull/63071)

## 사용 중단: `--moduleResolution node` (a.k.a. `--moduleResolution node10`)

`--moduleResolution node`는 Node.js 10의 동작을 가장 정확하게 반영하는 특정 버전의 Node.js 모듈 해석 알고리즘을 인코딩했습니다. 불행히도 이 타겟(및 해당 이름)은 이후 Node.js 해석 알고리즘의 많은 업데이트를 무시했으며, 현대 Node.js 버전의 동작을 더 이상 잘 나타내지 않습니다.

TypeScript 6.0에서는 `--moduleResolution node`(특히 `--moduleResolution node10`)가 사용 중단되었습니다. `--moduleResolution node`를 사용했던 사용자는 일반적으로 Node.js를 직접 타겟팅할 계획이라면 `--moduleResolution nodenext`로 마이그레이션하거나, 번들러 또는 Bun을 사용할 계획이라면 `--moduleResolution bundler`로 마이그레이션해야 합니다.

[이 이슈](https://github.com/microsoft/TypeScript/issues/62200)와 [해당 풀 요청](https://github.com/microsoft/TypeScript/pull/62338)에서 더 자세히 알아보세요.

## 사용 중단: `module`의 `amd`, `umd`, `systemjs` 값

다음 플래그 값은 더 이상 지원되지 않습니다.

*   `--module amd`
*   `--module umd`
*   `--module systemjs`
*   `--module none`

AMD, UMD 및 SystemJS는 브라우저에 네이티브 모듈 지원이 없었던 JavaScript 모듈의 초기 시대에 중요했습니다. "none"의 의미는 잘 정의되지 않았으며 종종 혼란을 야기했습니다. 오늘날 ESM은 브라우저와 Node.js에서 보편적으로 지원되며, import 맵과 번들러는 격차를 채우는 선호되는 방식이 되었습니다. 이러한 모듈 시스템을 여전히 타겟팅하는 경우, 적절한 ECMAScript 모듈 방출 타겟으로 마이그레이션하거나, 번들러 또는 다른 컴파일러를 채택하거나, 마이그레이션할 수 있을 때까지 TypeScript 5.x에 머무르는 것을 고려하십시오.

이는 `amd-module` 지시문의 지원 중단도 의미하며, 더 이상 효과가 없습니다.

[제안 이슈](https://github.com/microsoft/TypeScript/issues/62199) 및 [구현 풀 요청](https://github.com/microsoft/TypeScript/pull/62669)에서 더 자세히 알아보세요.

## 사용 중단: `--baseUrl`

`baseUrl` 옵션은 `paths`와 함께 가장 일반적으로 사용되며, 일반적으로 `paths`의 모든 값에 대한 접두사로 사용됩니다. 불행히도 `baseUrl`은 모듈 해석에 대한 조회 루트로도 간주됩니다.

예를 들어, 다음 `tsconfig.json`을 사용한다고 가정해 보겠습니다.

```
{
  "compilerOptions": {
    // ...
    "baseUrl": "./src",
    "paths": {
      "@app/*": ["app/*"],
      "@lib/*": ["lib/*"]
    }
  }
}

```

그리고 다음과 같은 임포트가 있습니다.

```
import * as someModule from "someModule.js";

```

TypeScript는 개발자가 `@app/` 및 `@lib/`로 시작하는 모듈에 대한 매핑만 추가하려고 했더라도 이 모듈을 `src/someModule.js`로 해석할 가능성이 높습니다.

최선의 경우, 이는 번들러가 무시할 "보기 좋지 않은" 경로로 이어지는 경우가 많았지만, 런타임에서 결코 작동하지 않았을 많은 임포트 경로가 TypeScript에 의해 "괜찮은" 것으로 간주되는 경우가 많았습니다.

`path` 매핑은 오랫동안 `baseUrl`을 지정할 필요가 없었으며, 실제로 `baseUrl`을 사용하는 대부분의 프로젝트는 `paths` 항목의 접두사로만 사용합니다. TypeScript 6.0에서는 `baseUrl`이 사용 중단되며 더 이상 모듈 해석에 대한 조회 루트로 간주되지 않습니다.

`baseUrl`을 경로 매핑 항목의 접두사로 사용했던 개발자는 단순히 `baseUrl`을 제거하고 `paths` 항목에 접두사를 추가할 수 있습니다.

```
  {
    "compilerOptions": {
      // ...
-     "baseUrl": "./src",
      "paths": {
-       "@app/*": ["app/*"],
-       "@lib/*": ["lib/*"]
+       "@app/*": ["./src/app/*"],
+       "@lib/*": ["./src/lib/*"]
      }
    }
  }

```

실제로 `baseUrl`을 조회 루트로 사용했던 개발자는 이전 동작을 유지하기 위해 명시적 경로 매핑을 추가할 수도 있습니다.

```
{
  "compilerOptions": {
    // ...
    "paths": {
      // baseUrl을 대체하는 새로운 와일드카드:
      "*": ["./src/*"],

      // 다른 모든 경로는 이제 명시적 공통 접두사를 갖습니다:
      "@app/*": ["./src/app/*"],
      "@lib/*": ["./src/lib/*"],
    }
  }
}

```

그러나 이는 극히 드뭅니다. 대부분의 개발자는 단순히 `baseUrl`을 제거하고 `paths` 항목에 적절한 접두사를 추가하는 것이 좋습니다.

[이 이슈](https://github.com/microsoft/TypeScript/issues/62207) 및 [해당 풀 요청](https://github.com/microsoft/TypeScript/pull/62509)에서 더 자세히 알아보세요.

## 사용 중단: `--moduleResolution classic`

`moduleResolution: classic` 설정이 제거되었습니다. `classic` 해석 전략은 TypeScript의 원래 모듈 해석 알고리즘이었으며 Node.js의 해석 알고리즘이 사실상의 표준이 되기 전에 나왔습니다. 오늘날 모든 실제 사용 사례는 `nodenext` 또는 `bundler`로 처리됩니다. `classic`을 사용했다면 이러한 최신 해석 전략 중 하나로 마이그레이션하십시오.

[이 이슈](https://github.com/microsoft/TypeScript/issues/62206) 및 [구현 풀 요청](https://github.com/microsoft/TypeScript/pull/62669)에서 더 자세히 알아보세요.

## 사용 중단: `--esModuleInterop false` 및 `--allowSyntheticDefaultImports false`

다음 설정은 `false`로 설정할 수 없습니다.

*   `esModuleInterop`
*   `allowSyntheticDefaultImports`

`esModuleInterop` 및 `allowSyntheticDefaultImports`는 원래 기존 프로젝트를 중단시키지 않기 위해 선택 사항이었습니다. 그러나 해당 기능이 지원하는 동작은 수년 동안 권장 기본값이었습니다. 이를 `false`로 설정하면 CommonJS 모듈을 ESM에서 소비할 때 미묘한 런타임 문제가 발생하는 경우가 많았습니다. TypeScript 6.0에서는 더 안전한 상호 운용 동작이 항상 활성화됩니다.

이전 동작에 의존하는 임포트가 있는 경우 조정해야 할 수 있습니다.

```
// 이전 (esModuleInterop: false)
import * as express from "express";

// 이후 (esModuleInterop 항상 활성화)
import express from "express";

```

[이 이슈](https://github.com/microsoft/TypeScript/issues/62529) 및 [해당 풀 요청](https://github.com/microsoft/TypeScript/pull/62567)에서 더 자세히 알아보세요.

## 사용 중단: `--alwaysStrict false`

`alwaysStrict` 플래그는 `"use strict";` 지시문의 추론 및 내보내기를 참조합니다. TypeScript 6.0에서는 모든 코드가 [JavaScript 엄격 모드](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Strict_mode)로 간주되며, 이는 가장 눈에 띄게 문법적 모서리 사례를 예약어에 대해 영향을 미치는 JS 의미 체계 세트입니다. "느슨한 모드" 코드가 `await`, `static`, `private` 또는 `public`과 같은 예약어를 일반 식별자로 사용하는 경우 이름을 바꿔야 합니다. 비엄격 모드 코드에서 `this`의 의미에 대한 미묘한 의미 체계에 의존했다면 코드도 조정해야 할 수 있습니다.

[이 이슈](https://github.com/microsoft/TypeScript/issues/62213) 및 [해당 풀 요청](https://github.com/microsoft/TypeScript/pull/63089)에서 더 자세히 알아보세요.

## 사용 중단: `outFile`

`--outFile` 옵션이 TypeScript 6.0에서 제거되었습니다. 이 옵션은 원래 여러 입력 파일을 단일 출력 파일로 연결하도록 설계되었습니다. 그러나 Webpack, Rollup, esbuild, Vite, Parcel 등과 같은 외부 번들러는 이제 이 작업을 더 빠르고 더 잘, 그리고 훨씬 더 많은 구성 가능성으로 수행합니다. 이 옵션을 제거하면 구현이 단순화되고 TypeScript가 가장 잘하는 작업, 즉 타입 검사 및 선언 내보내기에 집중할 수 있습니다. `--outFile`을 현재 사용하고 있다면 외부 번들러로 마이그레이션해야 합니다. 대부분의 최신 번들러는 기본적으로 훌륭한 TypeScript 지원을 제공합니다.

## 사용 중단: 네임스페이스에 대한 레거시 `module` 구문

TypeScript 초기 버전에서는 네임스페이스를 선언하기 위해 `module` 키워드를 사용했습니다.

```
// ❌ 사용 중단된 구문 - 이제 오류
module Foo {
    export const bar = 10;
}

```

이 구문은 나중에 `namespace` 키워드를 사용한 최신 기본 형태의 별칭으로 지정되었습니다.

```
// ✅ 올바른 구문
namespace Foo {
    export const bar = 10;
}

```

`namespace`가 도입되었을 때 `module` 구문은 단순히 권장되지 않았습니다. 몇 년 전 TypeScript 언어 서비스는 이 키워드를 사용 중단으로 표시하고 `namespace`를 제안했습니다.

TypeScript 6.0에서는 `namespace`가 예상되는 곳에서 `module`을 사용하는 것이 이제 엄격한 사용 중단입니다. `module` 블록은 레거시 TypeScript 구문과 충돌할 수 있는 잠재적인 ECMAScript 제안이기 때문에 이 변경이 필요합니다.

선언적 모듈 선언 형식은 완전히 지원됩니다.

```
// ✅ 여전히 완벽하게 작동합니다
declare module "some-module" {
    export function doSomething(): void;
}

```

자세한 내용은 [이슈](https://github.com/microsoft/TypeScript/issues/62211)와 해당 풀 요청([https://github.com/microsoft/TypeScript/pull/62876](https://github.com/microsoft/TypeScript/pull/62876))을 참조하십시오.

## 사용 중단: 임포트의 `asserts` 키워드

`asserts` 키워드는 import assertions 제안을 통해 JavaScript 언어에 제안되었지만, 결국 제안이 [import attributes 제안](https://github.com/tc39/proposal-import-attributes)으로 변경되었으며, 이는 `asserts` 대신 `with` 키워드를 사용합니다.

따라서 `asserts` 구문은 이제 TypeScript 6.0에서 사용 중단되었으며, 이를 사용하면 오류가 발생합니다.

```
// ❌ 사용 중단된 구문 - 이제 오류입니다.
import blob from "./blahb.json" asserts { type: "json" }
//                              ~~~~~~~
// 오류: Import assertions는 import attributes로 대체되었습니다. 'asserts' 대신 'with'를 사용하십시오.

```

대신 import attributes에 `with` 구문을 사용합니다.

```
// ✅ 새 import attributes 구문으로 작동합니다.
import blob from "./blahb.json" with { type: "json" }

```

[이 이슈](https://github.com/microsoft/TypeScript/issues/62210) 및 해당 풀 요청([https://github.com/microsoft/TypeScript/pull/63077](https://github.com/microsoft/TypeScript/pull/63077))에서 더 자세히 알아보세요.

## 사용 중단: `no-default-lib` 지시문

`/// <reference no-default-lib="true"/>` 지시문은 대부분 오해와 오용되었습니다. TypeScript 6.0에서는 이 지시문이 더 이상 지원되지 않습니다. 이를 사용했다면 대신 `--noLib` 또는 `--libReplacement`를 사용하는 것을 고려하십시오.

[여기서 더 자세히 알아보세요.](https://github.com/microsoft/TypeScript/issues/62209) [해당 풀 요청](https://github.com/microsoft/TypeScript/pull/62435)에서.

## `tsconfig.json`이 존재할 때 명령줄 파일 지정은 이제 오류입니다.

현재 `tsconfig.json`이 있는 폴더에서 `tsc foo.ts`를 실행하면 구성 파일이 완전히 무시됩니다. 이는 확인 및 내보내기 옵션이 입력 파일에 적용될 것으로 예상하는 경우 종종 매우 혼란스러웠습니다.

TypeScript 6.0에서는 `tsconfig.json`을 포함하는 디렉토리에서 파일 인수로 `tsc`를 실행하면 이 동작을 명시적으로 만들기 위해 오류가 발생합니다.

```
error TS5112: tsconfig.json이 존재하지만 파일이 commandline에 지정되면 로드되지 않습니다. 이 오류를 건너뛰려면 '--ignoreConfig'를 사용하십시오.

```

`tsconfig.json`을 무시하고 TypeScript 기본값으로 `foo.ts`만 컴파일하려는 경우 새 `--ignoreConfig` 플래그를 사용할 수 있습니다.

```
tsc --ignoreConfig foo.ts

```

[이 이슈](https://github.com/microsoft/TypeScript/issues/62197) 및 해당 풀 요청([https://github.com/microsoft/TypeScript/pull/62477](https://github.com/microsoft/TypeScript/pull/62477))에서 더 자세히 알아보세요.

# TypeScript 7.0 준비

TypeScript 6.0은 전환 릴리스로 설계되었습니다. TypeScript 6.0에서 사용 중단된 옵션은 `"ignoreDeprecations": "6.0"`이 설정된 경우 오류 없이 계속 작동하지만, 해당 옵션은 **TypeScript 7.0**(네이티브 TypeScript 포트)에서 _전체 제거_될 것입니다. TypeScript 6.0으로 업그레이드한 후 사용 중단 경고가 표시되는 경우, TypeScript 7.0을 채택하기 전에(또는 프로젝트에서 [네이티브 미리 보기](https://www.npmjs.com/package/@typescript/native-preview)를 시도하기 전에) 이러한 경고를 해결하는 것이 좋습니다.

일정에 관해서는 TypeScript 7.0이 TypeScript 6.0 직후에 출시될 것으로 예상됩니다. 이를 통해 연속성을 유지하면서 채택 중에 발견된 마이그레이션 문제에 대한 더 빠른 피드백 루프를 얻을 수 있습니다.

# 다음은 무엇인가요?

이 시점에서 TypeScript 6.0은 기능이 완료되었으며 컴파일러에 대한 중요한 버그 수정 외에는 거의 변경되지 않을 것으로 예상됩니다. 향후 몇 주 동안 6.0 브랜치에 보고된 문제를 해결하는 데 집중할 것이므로 RC를 사용해 보고 피드백을 공유해 주시기 바랍니다.

[npm](https://www.typescriptlang.org/docs/handbook/nightly-builds.html) 및 [Visual Studio Code](https://marketplace.visualstudio.com/items?itemName=ms-vscode.vscode-typescript-next)에서도 야간 빌드를 게시하며, 최근 수정된 문제에 대한 더 빠른 스냅샷을 제공할 수 있습니다.

또한 TypeScript 7.0 작업을 계속하고 있으며, [네이티브 미리 보기](https://www.npmjs.com/package/@typescript/native-preview)의 야간 빌드와 [VS Code 확장 프로그램](https://marketplace.visualstudio.com/items?itemName=TypeScriptTeam.native-preview)도 게시합니다. 6.0과 7.0 모두에 대한 피드백을 매우 감사하게 생각하며, 가능하다면 둘 다 사용해 보시기 바랍니다.

그러니 프로젝트에서 TypeScript 6.0 RC를 사용해 보고 의견을 알려주세요!
