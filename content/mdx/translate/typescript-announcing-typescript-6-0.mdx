> 원문: [Announcing TypeScript 6.0](https://devblogs.microsoft.com/typescript/announcing-typescript-6-0/) — TypeScript Blog

오늘 TypeScript 6.0의 출시를 발표하게 되어 매우 기쁩니다!

TypeScript에 익숙하지 않으시다면, JavaScript 위에 타입을 위한 문법을 추가하여 오류를 잡고 풍부한 에디터 도구를 제공하는 언어입니다. TypeScript와 시작 방법에 대한 자세한 내용은 [TypeScript 웹사이트](https://www.typescriptlang.org/)에서 확인하실 수 있습니다.

이미 이 언어에 익숙하시다면, npm을 통해 다음 명령어로 TypeScript 6.0을 설치할 수 있습니다:

```
npm install -D typescript

```

TypeScript 6.0은 현재 JavaScript 코드베이스를 기반으로 하는 마지막 릴리스라는 점에서 독특합니다. [작년에 발표한 바와 같이](https://devblogs.microsoft.com/typescript/typescript-native-port/) (그리고 [최근 업데이트 내용](https://devblogs.microsoft.com/typescript/progress-on-typescript-7-december-2025/)), Go로 작성된 새로운 TypeScript 컴파일러 및 언어 서비스 코드베이스 작업을 진행 중이며, 이는 네이티브 코드의 속도와 공유 메모리 멀티스레딩을 활용합니다. 이 새로운 코드베이스가 TypeScript 7.0 이후의 기반이 될 것입니다.

TypeScript 6.0은 TypeScript 5.9와 7.0 사이의 다리 역할을 합니다. 따라서 TypeScript 6.0의 대부분의 변경 사항은 TypeScript 7.0 채택을 위한 준비와 정렬을 돕기 위한 것입니다. 믿기 어려우시겠지만, TypeScript 7.0은 **실제로 완료가 매우 가까워졌습니다**. [Visual Studio Code에서 미리 사용해보거나](https://marketplace.visualstudio.com/items?itemName=TypeScriptTeam.native-preview) [npm에서 설치할 수 있습니다](https://npmx.dev/package/@typescript/native-preview). 실제로 TypeScript 6.0을 채택할 수 있다면, TypeScript 7.0의 네이티브 미리보기를 사용해 보시는 것을 권장합니다.

그렇다고 해서 TypeScript 6.0에 정렬만 위한 변경 사항이 없는 것은 아닙니다. 이 릴리스의 주요 내용 중 일부를 살펴본 후, 7.0에서 변경되는 내용과 준비 방법에 대해 더 자세히 알아보겠습니다.

# 베타 및 RC 이후의 새로운 기능은 무엇인가요?

TypeScript 6.0 베타 이후 몇 가지 주목할 만한 변경 사항이 있었습니다. 대부분 TypeScript 7.0의 동작과 일치시키기 위한 것입니다.

하나의 조정은 제네릭 호출, 특히 제네릭 JSX 표현식에서 발생하는 함수 표현식의 타입 검사입니다([이 pull request 참조](https://github.com/microsoft/TypeScript/pull/63163)). 이 변경은 기존 코드에서 더 많은 버그를 잡아낼 것이지만, 일부 제네릭 호출에는 명시적인 타입 인자가 필요할 수 있습니다.

또한, `import assertion` 구문(`import ... assert {...}`)의 사용 중단을 `import()` 호출([이 pull request 참조](https://github.com/microsoft/TypeScript/pull/63172))로 확장했습니다. 예를 들어 `import(..., { assert: {...}})`와 같은 형태입니다.

마지막으로, 최신 웹 표준을 반영하여 DOM 타입이 업데이트되었으며, Temporal API에 대한 일부 조정도 포함되었습니다.

# `this` 없는 함수의 컨텍스트 민감도 감소

매개변수에 명시적으로 타입이 지정되지 않은 경우, TypeScript는 예상되는 타입을 기반으로 하거나 동일한 함수 호출의 다른 인자를 통해 추론할 수 있습니다.

```
declare function callIt<T>(obj: {
    produce: (x: number) => T,
    consume: (y: T) => void,
}): void;

// 작동함, 문제 없음.
callIt({
    produce: (x: number) => x * 2,
    consume: y => y.toFixed(),
});

// 작동함, 속성이 뒤바뀌어도 문제 없음.
callIt({
    consume: y => y.toFixed(),
    produce: (x: number) => x * 2,
});

```

여기서 TypeScript는 속성 순서에 관계없이 `produce` 함수에서 추론된 `T`를 기반으로 `consume` 함수의 `y` 타입을 추론할 수 있습니다. 하지만 이러한 함수가 화살표 함수 구문 대신 _메서드 구문_을 사용하여 작성되었다면 어떻게 될까요?

```
declare function callIt<T>(obj: {
    produce: (x: number) => T,
    consume: (y: T) => void,
}): void;

// 잘 작동함, `x`는 number로 추론됨.
callIt({
    produce(x: number) { return x * 2; },
    consume(y) { return y.toFixed(); },
});

callIt({
    consume(y) { return y.toFixed(); },
    //                  ~
    // 오류: 'y'는 'unknown' 타입입니다.

    produce(x: number) { return x * 2; },
});

```

이상하게도 `callIt`의 두 번째 호출에서 TypeScript가 `consume` 메서드의 `y` 타입을 추론할 수 없기 때문에 오류가 발생합니다. 여기서 발생하는 문제는 TypeScript가 `T` 후보를 찾을 때, 매개변수에 명시적 타입이 없는 함수를 먼저 건너뛴다는 것입니다. 이는 특정 함수가 올바르게 검사되려면 `T`의 추론된 타입이 필요할 수 있기 때문입니다. 우리의 경우, `consume` 함수를 분석하려면 `T`의 타입을 알아야 합니다.

이러한 함수를 _컨텍스트 민감 함수_라고 합니다. 즉, 명시적 타입이 없는 매개변수를 가진 함수입니다. 결국 타입 시스템은 이러한 매개변수의 타입을 알아내야 하지만, 이는 제네릭 함수에서의 추론 방식과 약간 충돌합니다. 두 가지가 타입에 대해 서로 다른 방향으로 "당기기" 때문입니다.

```
function callFunc<T>(callback: (x: T) => void, value: T) {
    return callback(value);
}

callFunc(x => x.toFixed(), 42);
//       ^
// 여기서 `x`의 타입을 알아내야 하지만,
// 콜백을 검사하기 위해 `T`의 타입도 알아내야 합니다.

```

이를 해결하기 위해 TypeScript는 타입 인자 추론 중에 컨텍스트 민감 함수를 건너뛰고, 대신 다른 인자들을 먼저 검사하고 추론합니다. 컨텍스트 민감 함수를 건너뛰는 것이 작동하지 않으면, 추론은 인자 목록에서 왼쪽에서 오른쪽으로, 확인되지 않은 모든 인자를 계속 진행합니다. 바로 위 예제에서 TypeScript는 `T` 추론 중에 콜백을 건너뛰지만, 두 번째 인자인 `42`를 보고 `T`가 `number`라고 추론합니다. 그런 다음 콜백 검사로 돌아올 때 `(x: number) => void`라는 컨텍스트 타입을 가지게 되어 `x`도 `number`라고 추론할 수 있게 합니다.

그렇다면 이전 예제에서는 무슨 일이 일어나고 있었을까요?

```
// 화살표 구문 - 오류 없음.
callIt({
    consume: y => y.toFixed(),
    produce: (x: number) => x * 2,
});

// 메서드 구문 - 오류!
callIt({
    consume(y) { return y.toFixed(); },
    //                  ~
    // 오류: 'y'는 'unknown' 타입입니다.

    produce(x: number) { return x * 2; },
});

```

두 예제 모두에서 `produce`에는 명시적으로 타입이 지정된 `x` 매개변수가 있는 함수가 할당되어 있습니다. 동일하게 확인되어야 하지 않을까요?

이 문제는 미묘합니다. 대부분의 함수(메서드 구문을 사용하는 함수 등)는 암시적인 `this` 매개변수를 가지지만, 화살표 함수는 그렇지 않습니다. `this`의 모든 사용은 `T`의 타입에 대해 "당기는" 것을 필요로 할 수 있습니다. 예를 들어, 포함된 객체 리터럴의 타입은 결국 `T`를 사용하는 `consume`의 타입이 필요할 수 있습니다.

하지만 `this`를 사용하지 않습니다! 물론 런타임에 `this` 값이 있을 수 있지만, 사용되지 않습니다!

TypeScript 6.0은 함수가 컨텍스트 민감인지 여부를 결정할 때 이를 고려합니다. `this`가 함수 내에서 실제로 _사용되지_ 않는다면, 컨텍스트 민감으로 간주되지 않습니다. 이는 이러한 함수가 타입 추론에 대해 더 높은 우선순위를 가지게 되며, 위의 모든 예제가 이제 작동한다는 것을 의미합니다!

[이 변경 사항은](https://github.com/microsoft/TypeScript/pull/62243) [Mateusz Burzyński](https://github.com/Andarist)의 작업 덕분에 제공되었습니다.

# `#/`로 시작하는 하위 경로 가져오기

Node.js가 모듈 지원을 추가하면서 ["하위 경로 가져오기"](https://nodejs.org/api/packages.html#subpath-imports)라는 기능을 추가했습니다. 이것은 기본적으로 `imports`라는 필드로, 패키지가 패키지 내의 모듈에 대한 내부 별칭을 만들 수 있게 해줍니다.

```
{
    "name": "my-package",
    "type": "module",
    "imports": {
        "#root/*": "./dist/*"
    }
}

```

이를 통해 `my-package`의 모듈은 `#root/`로 시작하는 경로에서 가져올 수 있습니다.

```
import * as utils from "#root/utils.js";

```

다음과 같은 상대 경로를 사용하는 대신 말입니다.

```
import * as utils from "../../utils.js";

```

이 기능에 대한 한 가지 사소한 불편함은 개발자가 하위 경로 가져오기를 지정할 때 항상 `#` 뒤에 _무언가를_ 작성해야 한다는 것이었습니다. 여기서 `root`를 사용했지만, `./dist/` 외에는 매핑하는 디렉토리가 없기 때문에 약간 쓸모없습니다.

번들러를 사용한 개발자들도 긴 상대 경로를 피하기 위해 경로 매핑을 사용하는 데 익숙합니다. 번들러와 함께하는 일반적인 관례는 간단한 `@/`를 접두사로 사용하는 것이었습니다. 안타깝게도 하위 경로 가져오기는 `#/`로 시작할 수 없었기 때문에 프로젝트에서 채택하려는 개발자들에게 많은 혼란을 야기했습니다.

하지만 최근에 [Node.js가 `#/`로 시작하는 하위 경로 가져오기에 대한 지원을 추가했습니다](https://github.com/nodejs/node/pull/60864). 이를 통해 패키지는 추가 세그먼트를 추가할 필요 없이 하위 경로 가져오기에 간단한 `#/` 접두사를 사용할 수 있습니다.

```
{
    "name": "my-package",
    "type": "module",
    "imports": {
        "#/*": "./dist/*"
    }
}

```

이는 최신 Node.js 20 릴리스에서 지원되며, TypeScript는 이제 `--moduleResolution` 설정의 `nodenext` 및 `bundler` 옵션에서 이를 지원합니다.

이 작업은 [magic-akari](https://github.com/magic-akari) 덕분에 수행되었으며, [구현 pull request는 여기에서 찾을 수 있습니다](https://github.com/microsoft/TypeScript/pull/62844).

# `--moduleResolution bundler`와 `--module commonjs` 결합

TypeScript의 `--moduleResolution bundler` 설정은 이전에 `--module esnext` 또는 `--module preserve`와만 함께 사용할 수 있었습니다. 그러나 `--moduleResolution node` (일명 `--moduleResolution node10`)의 사용 중단으로 인해, 이 새로운 조합은 많은 프로젝트에 대한 가장 적합한 업그레이드 경로가 될 수 있습니다.

프로젝트는 종종 다음 중 하나로 마이그레이션을 계획하는 것을 선호할 것입니다:

*   `--module preserve` 및 `--moduleResolution bundler`
*   `--module nodenext`

프로젝트 유형(예: 번들 웹 앱, Bun 앱 또는 Node.js 앱)에 따라 달라집니다.

더 많은 정보는 [이 구현 pull request](https://github.com/microsoft/TypeScript/pull/62320)에서 확인할 수 있습니다.

# `--stableTypeOrdering` 플래그

[TypeScript의 네이티브 포트](https://devblogs.microsoft.com/typescript/typescript-native-port/)에 대한 지속적인 작업의 일환으로, 6.0에서 7.0로의 마이그레이션을 지원하기 위한 `--stableTypeOrdering`이라는 새로운 플래그를 도입했습니다.

현재 TypeScript는 타입이 발견되는 순서대로 타입 ID(내부 추적 번호)를 할당하고, 이러한 ID를 사용하여 union 타입을 일관되게 정렬합니다. 속성에도 유사한 과정이 적용됩니다. 결과적으로 프로그램에서 항목이 선언되는 순서는 선언 방출과 같은 사항에 예기치 않은 영향을 줄 수 있습니다.

예를 들어, 이 파일의 선언 방출을 고려해 보세요:

```
// 입력: some-file.ts
export function foo(condition: boolean) {
    return condition ? 100 : 500;
}

// 출력: some-file.d.ts
export declare function foo(condition: boolean): 100 | 500;
//                                               ^^^^^^^^^
//             이 union의 순서에 주목: 100, 그 다음 500.

```

`foo` _위에_ 관련 없는 `const`를 추가하면 선언 방출이 변경됩니다:

```
// 입력: some-file.ts
const x = 500;
export function foo(condition: boolean) {
    return condition ? 100 : 500;
}

// 출력: some-file.d.ts
export declare function foo(condition: boolean): 500 | 100;
//                                               ^^^^^^^^^
//                           여기서 순서 변경에 주목.

```

이는 리터럴 타입 `500`이 `const x` 선언을 분석할 때 먼저 처리되었기 때문에 `100`보다 낮은 타입 ID를 받기 때문에 발생합니다. 매우 드문 경우에 이 순서 변경은 프로그램 처리 순서에 따라 오류가 나타나거나 사라지게 할 수도 있지만, 일반적으로 이 순서에서 눈치챌 수 있는 주요 부분은 방출된 선언 파일 또는 에디터에서 타입이 표시되는 방식입니다.

TypeScript 7의 주요 아키텍처 개선 사항 중 하나는 병렬 타입 검사로, 전체 검사 시간을 크게 단축합니다. 그러나 병렬성은 문제를 야기합니다. 서로 다른 타입 검사기가 노드, 타입 및 심볼을 다른 순서로 방문할 때, 이러한 구성 요소에 할당되는 내부 ID는 결정적이지 않게 됩니다. 이는 결국 혼란스러운 비결정적 출력을 초래하며, 동일한 프로그램에서 동일한 내용을 가진 두 파일이 다른 선언 파일을 생성하거나 동일한 파일을 분석할 때 다른 오류를 계산할 수도 있습니다. 이를 해결하기 위해 TypeScript 7.0은 객체의 내용에 기반한 결정론적 알고리즘에 따라 내부 객체(예: 타입 및 심볼)를 정렬합니다. 이는 모든 검사기가 생성 시점과 방법에 관계없이 동일한 객체 순서를 접하도록 보장합니다. 결과적으로, 주어진 예에서 TypeScript는 _항상_ `100 | 500`을 출력하여 순서 불안정성을 완전히 제거합니다.

이는 TypeScript 6과 7이 때때로 다른 순서를 표시할 수 있음을 의미합니다. 이러한 순서 변경은 거의 항상 무해하지만, 실행 간 컴파일러 출력을 비교하는 경우(예: 6.0 대 7.0에서 방출된 선언 파일 확인), 이러한 다른 순서는 정확성을 평가하기 어렵게 만드는 많은 잡음을 발생시킬 수 있습니다. 하지만 때로는 순서 변경으로 인해 타입 오류가 나타나거나 사라지는 것을 볼 수 있으며, 이는 더욱 혼란스러울 수 있습니다.

이 상황을 돕기 위해 6.0에서는 새로운 `--stableTypeOrdering` 플래그를 지정할 수 있습니다. 이는 6.0의 타입 순서 동작을 7.0의 동작과 일치시켜 두 코드베이스 간의 차이점을 줄입니다. 이 플래그를 항상 사용하는 것이 좋지는 않습니다. 타입 검사에 상당한 지연을 줄 수 있기 때문입니다(코드베이스에 따라 최대 25%).

`--stableTypeOrdering`을 사용하여 타입 오류가 발생하는 경우, 이는 일반적으로 추론 차이 때문입니다. `--stableTypeOrdering`이 없는 이전 추론은 현재 프로그램의 타입 순서에 기반하여 작동했습니다. 이를 돕기 위해 종종 명시적인 타입을 제공하는 것이 도움이 됩니다.

```
- someFunctionCall(/*...*/);
+ someFunctionCall<SomeExplicitType>(/*...*/);

```

또는 호출에 전달하려는 인자에 대한 변수 주석입니다.

```
- const someVariable = { /*... 복잡한 객체 ...*/ };
+ const someVariable: SomeExplicitType = { /*... 복잡한 객체 ...*/ };

someFunctionCall(someVariable);

```

**이 플래그는 6.0과 7.0 간의 차이점을 진단하는 데만 사용되며, 장기적인 기능으로 사용하기 위한 것이 아닙니다.**

[이 pull-request에서 더 자세히 보기](https://github.com/microsoft/TypeScript/pull/63084).

# `target` 및 `lib`를 위한 `es2025` 옵션

TypeScript 6.0은 `target` 및 `lib` 모두에 대해 `es2025` 옵션을 지원합니다. ES2025에는 새로운 JavaScript 언어 기능이 없지만, 이 새로운 타겟은 빌트인 API(예: `RegExp.escape`)에 대한 새로운 타입을 추가하고 몇 가지 선언을 `esnext`에서 `es2025`로 이동합니다(예: `Promise.try`, `Iterator` 메서드 및 `Set` 메서드). [새로운 타겟을 활성화하기 위한 작업](https://github.com/microsoft/TypeScript/pull/63046)은 [Kenta Moriuchi](https://github.com/petamoriken)의 기여 덕분에 이루어졌습니다.

# `Temporal`을 위한 새 타입

오랫동안 기다려온 [Temporal 제안](https://github.com/tc39/proposal-temporal)이 스테이지 4에 도달했으며 향후 ECMAScript 표준의 일부가 될 것입니다. TypeScript 6.0에는 Temporal API에 대한 빌트인 타입이 포함되어 있으므로 `--target esnext` 또는 `"lib": ["esnext"]` (또는 더 세분화된 `esnext.temporal`)를 통해 오늘날 TypeScript 코드에서 사용할 수 있습니다.

```
let yesterday = Temporal.Now.instant().subtract({
    hours: 24,
});

let tomorrow = Temporal.Now.instant().add({
    hours: 24,
});

console.log(`Yesterday: ${yesterday}`);
console.log(`Tomorrow: ${tomorrow}`);

```

Temporal은 이미 여러 런타임에서 사용할 수 있으며, 스테이지 4 상태로 JavaScript 언어의 공식 부분이 되었습니다. [Temporal API에 대한 문서는 MDN에서 확인할 수 있습니다](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Temporal).

[이 작업은](https://github.com/microsoft/TypeScript/pull/62628) GitHub 사용자 [Renegade334](https://github.com/Renegade334)의 기여 덕분에 이루어졌습니다.

# "upsert" 메서드 (일명 `getOrInsert`)를 위한 새 타입

`Map`을 사용하는 일반적인 패턴은 키가 존재하는지 확인하고, 존재하지 않으면 기본값을 설정하고 가져오는 것입니다.

```
function processOptions(compilerOptions: Map<string, unknown>) {
    let strictValue: unknown;
    if (compilerOptions.has("strict")) {
        strictValue = compilerOptions.get("strict");
    }
    else {
        strictValue = true;
        compilerOptions.set("strict", strictValue);
    }
    // ...
}

```

이 패턴은 번거로울 수 있습니다. [ECMAScript의 "upsert" 제안](https://github.com/tc39/proposal-upsert)이 최근 스테이지 4에 도달했으며, `Map` 및 `WeakMap`에 두 개의 새로운 메서드를 도입했습니다:

*   `getOrInsert`
*   `getOrInsertComputed`

이 메서드는 `esnext` 라이브러리에 추가되어 TypeScript 6.0에서 즉시 사용할 수 있습니다.

`getOrInsert`를 사용하면 위의 코드를 다음과 같이 바꿀 수 있습니다:

```
function processOptions(compilerOptions: Map<string, unknown>) {
    let strictValue = compilerOptions.getOrInsert("strict", true);
    // ...
}

```

`getOrInsertComputed`는 유사하게 작동하지만, 기본값을 계산하기에 비용이 많이 드는 경우(예: 많은 계산, 할당이 필요하거나 오래 실행되는 동기 I/O를 수행하는 경우)에 사용됩니다. 대신, 키가 아직 존재하지 않는 경우에만 호출되는 콜백을 받습니다.

```
someMap.getOrInsertComputed("someKey", () => {
    return computeSomeExpensiveValue(/*...*/);
});

```

이 콜백은 키를 인자로 받기도 하는데, 이는 기본값이 키에 기반하는 경우 유용할 수 있습니다.

```
someMap.getOrInsertComputed(someKey, computeSomeExpensiveDefaultValue);

function computeSomeExpensiveValue(key: string) {
    // ...
}

```

[이 업데이트는](https://github.com/microsoft/TypeScript/pull/62612) GitHub 사용자 [Renegade334](https://github.com/Renegade334)의 기여 덕분에 이루어졌습니다.

# `RegExp.escape`

정규 표현식 내에서 일치시킬 문자열 리터럴을 구성할 때, `*`, `+`, `?`, `(`, `)` 등과 같은 특수 정규 표현식 문자를 이스케이프하는 것이 중요합니다. [RegExp Escaping ECMAScript 제안](https://github.com/tc39/proposal-regex-escaping)이 스테이지 4에 도달했으며, 이를 처리해 주는 새로운 `RegExp.escape` 함수를 도입했습니다.

```
function matchWholeWord(word: string, text: string) {
    const escapedWord = RegExp.escape(word);
    const regex = new RegExp(`\\b${escapedWord}\\b`, "g");
    return text.match(regex);
}

```

`RegExp.escape`는 `es2025` 라이브러리에서 사용할 수 있으므로 TypeScript 6.0에서 오늘부터 사용할 수 있습니다.

[이 작업은](https://github.com/microsoft/TypeScript/pull/63046) [Kenta Moriuchi](https://github.com/petamoriken)의 기여 덕분에 이루어졌습니다.

# `dom` 라이브러리에 `dom.iterable` 및 `dom.asynciterable` 포함

TypeScript의 `lib` 옵션을 사용하면 대상 런타임이 가지고 있는 전역 선언을 지정할 수 있습니다. 하나의 옵션은 웹 환경(즉, DOM API를 구현하는 브라우저)을 나타내는 `dom`입니다. 이전에 DOM API는 `Iterable` 및 `AsyncIterable`을 지원하지 않는 환경에 대해 `dom.iterable` 및 `dom.asynciterable`로 부분적으로 분할되었습니다. 이는 `NodeList` 또는 `HTMLCollection`과 같은 DOM 컬렉션의 반복 메서드를 사용하려면 `dom.iterable`을 명시적으로 추가해야 함을 의미했습니다.

TypeScript 6.0에서는 `lib.dom.iterable.d.ts` 및 `lib.dom.asynciterable.d.ts`의 내용이 `lib.dom.d.ts`에 완전히 포함되었습니다. 구성 파일의 `"lib"` 배열에 `dom.iterable` 및 `dom.asynciterable`을 계속 참조할 수 있지만, 이제는 빈 파일입니다.

```
// TypeScript 6.0 이전에는 "lib": ["dom", "dom.iterable"]이 필요했습니다.
// 이제는 "lib": ["dom"]만으로 작동합니다.
for (const element of document.querySelectorAll("div")) {
    console.log(element.textContent);
}

```

이는 일반적인 혼란 지점을 제거하는 품질 향상 기능이며, 대부분의 최신 브라우저는 이러한 기능을 지원하지 않습니다. 이미 `dom`과 `dom.iterable`을 모두 포함하고 있었다면, 이제 `dom`만으로 단순화할 수 있습니다.

[이 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/60959) 및 해당 [pull request](https://github.com/microsoft/TypeScript/pull/62111).

# TypeScript 6.0의 호환성 변경 및 사용 중단

TypeScript 6.0은 TypeScript 컴파일러의 네이티브 포트인 TypeScript 7.0을 준비하기 위한 중요한 전환 릴리스로 출시됩니다. TypeScript 6.0은 기존 TypeScript 지식과의 완전한 호환성을 유지하고 TypeScript 5.9와의 API 호환성을 계속 유지하지만, 이 릴리스는 진화하는 JavaScript 생태계를 반영하고 TypeScript 7.0을 위한 기반을 마련하는 여러 호환성 변경 및 사용 중단 사항을 도입합니다.

TypeScript 5.0 이후 2년 동안 개발자가 JavaScript를 작성하고 배포하는 방식에 지속적인 변화가 있었습니다:

*   사실상 모든 런타임 환경이 이제 "evergreen"입니다. 진정한 레거시 환경(ES5)은 극히 드뭅니다.
*   번들러 및 ESM은 새로운 프로젝트에 대한 가장 일반적인 모듈 대상이 되었지만, CommonJS는 여전히 중요한 대상입니다. AMD 및 기타 브라우저 사용자 영역 모듈 시스템은 2012년보다 훨씬 드뭅니다.
*   거의 모든 패키지는 모듈 시스템을 통해 소비될 수 있습니다. UMD 패키지는 여전히 존재하지만, 거의 새로운 코드가 전역 변수로만 제공되지는 않습니다.
*   `tsconfig.json`은 거의 보편적인 구성 메커니즘입니다.
*   "더 엄격한" 타입에 대한 요구는 계속 증가하고 있습니다.
*   TypeScript 빌드 성능이 최우선입니다. TypeScript 7의 개선 사항에도 불구하고 성능은 항상 주요 목표여야 하며, 성능이 지원되지 않는 옵션은 더 강력하게 정당화되어야 합니다.

따라서 TypeScript 6.0 및 7.0은 이러한 현실을 염두에 두고 설계되었습니다. TypeScript 6.0의 경우, 이러한 사용 중단은 `tsconfig`에서 `"ignoreDeprecations": "6.0"`을 설정하여 무시할 수 있습니다. 그러나 TypeScript 7.0은 이러한 사용 중단된 옵션을 _지원하지 않을_ 것임을 유의하십시오.

몇 가지 필요한 조정은 코딩 모드 또는 도구로 자동 수행될 수 있습니다. 예를 들어, [실험적인 `ts5to6` 도구](https://github.com/andrewbranch/ts5to6)는 코드베이스 전체의 `baseUrl` 및 `rootDir`를 자동으로 조정할 수 있습니다.

## 사전 조정

아래에서 특정 조정을 다룰 것이지만, 일부 사용 중단 및 동작 변경에는 문제가 있는 근본 원인을 직접적으로 가리키는 오류 메시지가 항상 있는 것은 아님을 참고해야 합니다. 따라서 **많은 프로젝트에서 다음 중 하나 이상이 필요할 것입니다**:

*   `tsconfig`에서 `"types"` 배열을 설정합니다. 일반적으로 `"types": ["node"]`로 설정합니다.
    
    `"types": ["*"]`는 5.9 동작을 복원하지만, 빌드 성능 및 예측 가능성을 향상시키기 위해 명시적 배열을 사용하는 것이 좋습니다.
    
    식별자 누락 또는 확인되지 않은 빌트인 모듈과 관련된 _많은_ 타입 오류가 발생하는 경우 이 문제가 발생한 것임을 알 수 있습니다.
    
*   이전 이 추론에 의존했다면 `"rootDir": "./src"`를 설정합니다.
    
    `./dist/src/index.js` 대신 `./dist/index.js`에 파일이 기록되는 것을 보는 경우 종종 이 문제가 발생했음을 알 수 있습니다.
    

## 간단한 기본값 변경

여러 컴파일러 옵션의 기본값이 이제 최신 개발 관행을 더 잘 반영하도록 업데이트되었습니다.

*   **`strict`가 이제 기본값으로 `true`입니다**: 더 엄격한 타입에 대한 요구는 계속 증가하고 있으며, 대부분의 새로운 프로젝트에서 `strict` 모드를 활성화하기를 원한다는 것을 알게 되었습니다. 이미 `"strict": true`를 사용하고 있었다면 아무것도 변경되지 않습니다. 이전 기본값인 `false`에 의존했다면 `tsconfig.json`에서 `"strict": false`를 명시적으로 설정해야 합니다.
    
*   **`module`이 `esnext`로 기본 설정됩니다**: 마찬가지로, 새로운 기본값 `module`은 `esnext`이며, ESM이 이제 지배적인 모듈 형식임을 인식합니다.
    
*   **`target`이 현재 연도 ES 버전으로 기본 설정됩니다**: 새로운 기본 `target`은 지원되는 가장 최신 ECMAScript 사양 버전입니다(실질적으로 부동 타겟). 현재 해당 타겟은 `es2025`입니다. 이는 대부분의 개발자가 evergreen 런타임으로 배포하고 있으며 이전 ECMAScript 버전으로 트랜스파일할 필요가 없다는 현실을 반영합니다.
    
*   **`noUncheckedSideEffectImports`가 이제 기본값으로 `true`입니다**: 이는 부작용 전용 가져오기에서의 오타로 인한 문제를 포착하는 데 도움이 됩니다.
    
*   **`libReplacement`이 이제 기본값으로 `false`입니다**: 이 플래그는 이전에는 실행마다 많은 모듈 해결 실패를 발생시켰으며, 이는 `--watch` 및 에디터 시나리오에서 추적해야 하는 위치 수를 늘렸습니다. 새 프로젝트에서 `libReplacement`는 다른 명시적 구성이 수행될 때까지 아무 작업도 수행하지 않으므로, 기본적으로 성능 향상을 위해 기본적으로 해제하는 것이 좋습니다.
    

이러한 새 기본값이 프로젝트에 문제가 되는 경우 `tsconfig.json`에서 이전 값을 명시적으로 지정할 수 있습니다.

## `rootDir`이 이제 `.`으로 기본 설정됩니다

`rootDir`은 출력 디렉토리에 상대적인 출력 파일의 디렉토리 구조를 제어합니다. 이전에는 `rootDir`을 지정하지 않은 경우, 선언 파일이 아닌 모든 입력 파일의 공통 디렉토리를 기반으로 추론되었습니다. 그러나 종종 파일이 프로젝트에 속하는지 여부를 해당 프로젝트를 로드하고 구문 분석하려고 시도하기 전에는 알 수 없다는 것을 의미했습니다. 또한 TypeScript가 프로그램의 모든 파일 경로를 분석하여 해당 공통 소스 디렉토리를 추론하는 데 더 많은 시간을 소비해야 했습니다.

TypeScript 6.0에서는 기본 `rootDir`이 항상 `tsconfig.json` 파일을 포함하는 디렉토리가 됩니다. `rootDir`은 `tsconfig.json` 파일 없이 명령줄에서 `tsc`를 사용할 때만 추론됩니다.

`tsconfig.json` 디렉토리보다 더 깊은 수준에 소스 파일이 있고 TypeScript가 소스 파일의 공통 루트 디렉토리를 추론하도록 의존했다면 `rootDir`을 명시적으로 설정해야 합니다:

```
  {
      "compilerOptions": {
          // ...
+         "rootDir": "./src"
      },
      "include": ["./src"]
  }

```

마찬가지로 `tsconfig.json`이 포함된 `tsconfig.json` 외부의 파일을 참조하는 경우 `rootDir`을 조정하여 해당 파일을 포함해야 합니다.

```
  {
      "compilerOptions": {
          // ...
+         "rootDir": "../src"
      },
      "include": ["../src/**/*.tests.ts"]
  }

```

[이 논의에서 더 보기](https://github.com/microsoft/TypeScript/issues/62194) 및 [여기서의 구현](https://github.com/microsoft/TypeScript/pull/62418).

## `types`가 이제 `[]`으로 기본 설정됩니다

`tsconfig.json`에서 `compilerOptions`의 `types` 필드는 컴파일 중에 전역 범위에 포함될 패키지 이름 목록을 지정합니다. 일반적으로 `node_modules`의 패키지는 소스 코드의 가져오기를 통해 자동으로 포함됩니다. 그러나 편의를 위해 TypeScript는 기본적으로 `@types`의 모든 패키지를 포함하여, 직접 가져올 필요 없이 `@types/node`의 `process` 또는 `"fs"` 모듈과 같은 전역 선언이나 `@types/jest`의 `describe` 및 `it`과 같은 것을 얻을 수 있습니다.

어떤 의미에서 `types` 값은 이전에는 "`node_modules/@types`의 모든 것을 열거"하는 것으로 기본 설정되었습니다. 이는 정상적인 리포지토리 설정에 오늘날 수백 개의 `@types` 패키지가 포함될 수 있으므로 (특히 평탄화된 `node_modules`를 가진 다중 프로젝트 워크스페이스에서는) _매우_ 비용이 많이 들 수 있습니다. 최신 프로젝트는 거의 항상 `@types/node`, `@types/jest` 또는 몇 가지 다른 일반적인 전역 영향을 미치는 패키지만 필요로 합니다.

TypeScript 6.0에서는 기본 `types` 값이 `[]` (빈 배열)가 됩니다. 이 변경은 프로젝트가 빌드 시점에 의도하지 않게 수백 또는 수천 개의 불필요한 선언 파일을 가져오는 것을 방지합니다. 우리가 살펴본 많은 프로젝트에서 `types`를 적절하게 설정하기만 해도 빌드 시간이 20-50% 향상되었습니다.

**이는 많은 프로젝트에 영향을 미칩니다.** `"types": ["node"]` 또는 몇 가지 다른 것을 추가해야 할 가능성이 높습니다:

```
  {
      "compilerOptions": {
          // 필요한 @types 패키지를 명시적으로 나열합니다.
+         "types": ["node", "jest"]
      }
  }

```

`*` 항목을 지정하여 이전 열거 동작을 다시 활성화할 수도 있습니다:

```
  {
      "compilerOptions": {
          // 모든 타입을 로드합니다 - TypeScript 5.9 이전의 기본값입니다.
+         "types": ["*"]
      }
  }

```

다음과 같은 새로운 오류 메시지가 발생하는 경우:

```
모듈 '...' 또는 해당 타입 선언을 찾을 수 없습니다.
'fs' 이름을 찾을 수 없습니다. node에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/node`를 시도한 다음 tsconfig의 types 필드에 'node'를 추가하십시오.
'path' 이름을 찾을 수 없습니다. node에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/node`를 시도한 다음 tsconfig의 types 필드에 'node'를 추가하십시오.
'process' 이름을 찾을 수 없습니다. node에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/node`를 시도한 다음 tsconfig의 types 필드에 'node'를 추가하십시오.
'Bun' 이름을 찾을 수 없습니다. Bun에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/bun`을 시도한 다음 tsconfig의 types 필드에 'bun'을 추가하십시오.
'describe' 이름을 찾을 수 없습니다. 테스트 실행기에 대한 타입 정의를 설치해야 합니까? `npm i --save-dev @types/jest` 또는 `npm i --save-dev @types/mocha`를 시도한 다음 tsconfig의 types 필드에 'jest' 또는 'mocha'를 추가하십시오.

```

`types` 필드에 일부 항목을 추가해야 할 가능성이 높습니다.

[이 제안에서 더 보기](https://github.com/microsoft/TypeScript/issues/62195)와 [구현 pull request](https://github.com/microsoft/TypeScript/pull/63054) 함께.

## 사용 중단: `target: es5`

ECMAScript 5 타겟은 레거시 브라우저를 지원하는 데 오랫동안 중요했지만, 그 후속인 ECMAScript 2015(ES6)은 10년 이상 전에 출시되었으며 모든 최신 브라우저는 수년 동안 이를 지원해 왔습니다. Internet Explorer의 지원 종료와 evergreen 브라우저의 보편화로 인해 오늘날 ES5 출력에 대한 사용 사례는 매우 드뭅니다.

TypeScript의 가장 낮은 타겟은 이제 ES2015이며, `target: es5` 옵션은 사용 중단되었습니다. `target: es5`를 사용하고 있었다면, 더 최신 타겟으로 마이그레이션하거나 외부 컴파일러를 사용해야 합니다. 여전히 ES5 출력이 필요한 경우, TypeScript 소스를 직접 컴파일하거나 TypeScript 출력을 후처리하기 위해 외부 컴파일러를 사용하는 것이 좋습니다.

[이 사용 중단에 대한 자세한 내용은 여기에서 확인하십시오](https://github.com/microsoft/TypeScript/issues/62196)와 [구현 pull request](https://github.com/microsoft/TypeScript/pull/63067) 함께.

## 사용 중단: `--downlevelIteration`

`--downlevelIteration`은 ES5 방출에만 영향을 미치며, `--target es5`가 사용 중단되었기 때문에 `--downlevelIteration`은 더 이상 목적이 없습니다.

미묘하게도, `--target es2015`와 함께 `--downlevelIteration false`를 사용하는 것은 TypeScript 5.9 이전에서는 오류가 발생하지 않았습니다 (효과가 없었음에도 불구하고). TypeScript 6.0에서는 `--downlevelIteration`을 설정하면 사용 중단 오류가 발생합니다.

[여기서 구현 보기](https://github.com/microsoft/TypeScript/pull/63071).

## 사용 중단: `--moduleResolution node` (일명 `--moduleResolution node10`)

`--moduleResolution node`는 Node.js 10의 동작을 가장 정확하게 반영하는 특정 버전의 Node.js 모듈 확인 알고리즘을 인코딩했습니다. 안타깝게도 이 타겟(및 그 이름)은 그 이후로 발생한 Node.js 해결 알고리즘의 많은 업데이트를 무시하며, 더 이상 최신 Node.js 버전의 동작을 잘 나타내지 못합니다.

TypeScript 6.0에서는 `--moduleResolution node` (특히 `--moduleResolution node10`)가 사용 중단되었습니다. `--moduleResolution node`를 사용했던 사용자는 일반적으로 Node.js를 직접 대상으로 하는 경우 `--moduleResolution nodenext`로 마이그레이션하거나, 번들러 또는 Bun을 사용하는 경우 `--moduleResolution bundler`로 마이그레이션해야 합니다.

[이 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/62200) 및 [해당 pull request](https://github.com/microsoft/TypeScript/pull/62338).

## 사용 중단: `module`의 `amd`, `umd`, `systemjs` 값

다음 플래그 값은 더 이상 지원되지 않습니다.

*   `--module amd`
*   `--module umd`
*   `--module systemjs`
*   `--module none`

AMD, UMD 및 SystemJS는 브라우저가 네이티브 모듈 지원이 없었던 JavaScript 모듈 초기 시대에 중요했습니다. "none"의 의미론은 잘 정의되지 않아 혼란을 야기하는 경우가 많았습니다. 오늘날 ESM은 브라우저와 Node.js에서 보편적으로 지원되며, import map과 번들러 모두 격차를 메우는 데 선호되는 방법이 되었습니다. 이러한 모듈 시스템을 여전히 대상으로 하는 경우, 적절한 ECMAScript 모듈 방출 타겟으로 마이그레이션하거나, 번들러 또는 다른 컴파일러를 채택하거나, 마이그레이션할 수 있을 때까지 TypeScript 5.x를 유지하는 것을 고려하십시오.

이는 또한 `amd-module` 지시문의 지원이 중단됨을 의미하며, 더 이상 효과가 없습니다.

[이 제안 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/62199)와 [구현 pull request](https://github.com/microsoft/TypeScript/pull/62669) 함께.

## 사용 중단: `--baseUrl`

`baseUrl` 옵션은 `paths`와 함께 가장 일반적으로 사용되며, 일반적으로 `paths`의 모든 값에 대한 접두사로 사용됩니다. 안타깝게도 `baseUrl`은 모듈 확인을 위한 조회 루트로도 간주됩니다.

예를 들어, 다음 `tsconfig.json`을 사용합니다.

```
{
  "compilerOptions": {
    // ...
    "baseUrl": "./src",
    "paths": {
      "@app/*": ["app/*"],
      "@lib/*": ["lib/*"]
    }
  }
}

```

그리고 다음과 같은 가져오기:

```
import * as someModule from "someModule.js";

```

TypeScript는 개발자가 `@app/` 및 `@lib/`로 시작하는 모듈에 대한 매핑만 추가하려고 했더라도 이를 `src/someModule.js`로 해석할 것입니다.

최상의 경우, 이는 번들러가 무시할 "보기 좋지 않은" 경로를 종종 초래했지만, 런타임에 결코 작동하지 않을 많은 가져오기 경로를 TypeScript가 "괜찮다"고 간주하게 만들었습니다.

`path` 매핑은 오랫동안 `baseUrl`을 지정할 필요가 없었으며, 실제로는 `baseUrl`을 사용하는 대부분의 프로젝트는 `paths` 항목의 접두사로만 사용합니다. TypeScript 6.0에서는 `baseUrl`이 사용 중단되며 더 이상 모듈 확인을 위한 조회 루트로 간주되지 않습니다.

경로 매핑 항목에 대한 접두사로 `baseUrl`을 사용했던 개발자는 `baseUrl`을 제거하고 `paths` 항목에 접두사를 추가하기만 하면 됩니다:

```
  {
    "compilerOptions": {
      // ...
-     "baseUrl": "./src",
      "paths": {
-       "@app/*": ["app/*"],
-       "@lib/*": ["lib/*"]
+       "@app/*": ["./src/app/*"],
+       "@lib/*": ["./src/lib/*"]
      }
    }
  }

```

실제로 `baseUrl`을 조회 루트로 사용했던 개발자는 이전 동작을 유지하기 위해 명시적인 경로 매핑을 추가할 수도 있습니다:

```
{
  "compilerOptions": {
    // ...
    "paths": {
      // baseUrl을 대체하는 새로운 와일드카드:
      "*": ["./src/*"],

      // 다른 모든 경로는 이제 명시적인 공통 접두사를 가집니다:
      "@app/*": ["./src/app/*"],
      "@lib/*": ["./src/lib/*"],
    }
  }
}

```

그러나 이는 극히 드뭅니다. 대부분의 개발자는 `baseUrl`을 제거하고 `paths` 항목에 적절한 접두사를 추가하는 것이 좋습니다.

[이 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/62207) 및 [해당 pull request](https://github.com/microsoft/TypeScript/pull/62509).

## 사용 중단: `--moduleResolution classic`

`moduleResolution: classic` 설정이 제거되었습니다. `classic` 해석 전략은 TypeScript의 원래 모듈 해석 알고리즘이었으며 Node.js의 해석 알고리즘이 사실상의 표준이 되기 전에 만들어졌습니다. 오늘날 모든 실질적인 사용 사례는 `nodenext` 또는 `bundler`로 충족됩니다. `classic`을 사용하고 있었다면 이러한 최신 해석 전략으로 마이그레이션하십시오.

[이 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/62206) 및 [구현 pull request](https://github.com/microsoft/TypeScript/pull/62669).

## 사용 중단: `--esModuleInterop false` 및 `--allowSyntheticDefaultImports false`

다음 설정은 더 이상 `false`로 설정할 수 없습니다:

*   `esModuleInterop`
*   `allowSyntheticDefaultImports`

`esModuleInterop` 및 `allowSyntheticDefaultImports`는 원래 기존 프로젝트의 호환성을 위해 옵트인되었습니다. 그러나 해당 기능이 가능하게 하는 동작은 수년 동안 권장 기본값이었습니다. 이를 `false`로 설정하면 ESM에서 CommonJS 모듈을 소비할 때 미묘한 런타임 문제가 발생하는 경우가 많았습니다. TypeScript 6.0에서는 더 안전한 상호 운용 동작이 항상 활성화됩니다.

이전 동작에 의존하는 가져오기가 있는 경우 조정해야 할 수 있습니다:

```
// 이전 (esModuleInterop: false)
import * as express from "express";

// 이후 (esModuleInterop가 항상 활성화됨)
import express from "express";

```

[이 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/62529) 및 [구현 pull request](https://github.com/microsoft/TypeScript/pull/62567).

## 사용 중단: `--alwaysStrict false`

`alwaysStrict` 플래그는 `"use strict";` 지시문의 추론 및 방출을 참조합니다. TypeScript 6.0에서는 모든 코드가 [JavaScript strict mode](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Strict_mode)에 있는 것으로 간주됩니다. 이는 구문적 모서리 사례에 가장 눈에 띄게 영향을 미치는 일련의 JS 의미론입니다. 예약어(`await`, `static`, `private`, `public`)를 일반 식별자로 사용하는 "느슨한 모드" 코드가 있는 경우 이름을 바꿔야 합니다. 비엄격 코드에서 `this`의 의미에 대한 미묘한 의미론에 의존했다면 코드도 조정해야 할 수 있습니다.

[이 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/62213) 및 [해당 pull request](https://github.com/microsoft/TypeScript/pull/63089).

## 사용 중단: `outFile`

`--outFile` 옵션은 TypeScript 6.0에서 제거되었습니다. 이 옵션은 원래 여러 입력 파일을 단일 출력 파일로 연결하도록 설계되었습니다. 그러나 Webpack, Rollup, esbuild, Vite, Parcel 등과 같은 외부 번들러는 이제 더 빠르고, 더 좋고, 훨씬 더 많은 구성 가능성으로 이 작업을 수행합니다. 이 옵션을 제거하면 구현이 단순화되고 TypeScript가 가장 잘하는 것(타입 검사 및 선언 방출)에 집중할 수 있습니다. 현재 `--outFile`을 사용하고 있다면 외부 번들러로 마이그레이션해야 합니다. 대부분의 최신 번들러는 기본적으로 훌륭한 TypeScript 지원을 제공합니다.

## 사용 중단: 네임스페이스에 대한 레거시 `module` 구문

TypeScript의 초기 버전은 네임스페이스를 선언하기 위해 `module` 키워드를 사용했습니다:

```
// ❌ 사용 중단된 구문 - 이제 오류
module Foo {
    export const bar = 10;
}

```

이 구문은 나중에 `namespace` 키워드를 사용한 현대적인 선호 형식으로 별칭이 지정되었습니다:

```
// ✅ 올바른 구문
namespace Foo {
    export const bar = 10;
}

```

`namespace`가 도입되었을 때 `module` 구문은 단순히 권장되지 않았습니다. 몇 년 전에 TypeScript 언어 서비스는 `namespace`를 대신 제안하면서 해당 키워드를 사용 중단으로 표시하기 시작했습니다.

TypeScript 6.0에서는 `namespace`가 예상되는 곳에서 `module`을 사용하는 것이 이제 엄격하게 사용 중단되었습니다. `module` 블록은 레거시 TypeScript 구문과 충돌할 수 있는 잠재적인 ECMAScript 제안이기 때문에 이 변경이 필요합니다.

Ambient 모듈 선언 형식은 완전히 지원됩니다:

```
// ✅ 완벽하게 작동합니다.
declare module "some-module" {
    export function doSomething(): void;
}

```

[자세한 내용은 이 이슈](https://github.com/microsoft/TypeScript/issues/62211) 및 해당 [pull request](https://github.com/microsoft/TypeScript/pull/62876)에서 확인하십시오.

## 사용 중단: Imports의 `asserts` 키워드

`asserts` 키워드는 import assertions 제안을 통해 JavaScript 언어로 제안되었습니다. 그러나 해당 제안은 결국 `asserts` 대신 `with` 키워드를 사용하는 [import attributes 제안](https://github.com/tc39/proposal-import-attributes)으로 변경되었습니다.

따라서 `asserts` 구문은 TypeScript 6.0에서 사용 중단되었으며, 이를 사용하면 오류가 발생합니다:

```
// ❌ 사용 중단된 구문 - 이제 오류.
import blob from "./blahb.json" asserts { type: "json" }
//                              ~~~~~~~
// 오류: Import assertions는 import attributes로 대체되었습니다. 'asserts' 대신 'with'을 사용하십시오.

```

대신 import attributes에 대한 `with` 구문을 사용하십시오:

```
// ✅ 새로운 import attributes 구문으로 작동합니다.
import blob from "./blahb.json" with { type: "json" }

```

[이 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/62210) 및 해당 [pull request](https://github.com/microsoft/TypeScript/pull/63077).

## 사용 중단: `no-default-lib` 지시문

`/// <reference no-default-lib="true"/>` 지시문은 대부분 오해되고 오용되었습니다. TypeScript 6.0에서는 이 지시문이 더 이상 지원되지 않습니다. 사용하고 있었다면 `--noLib` 또는 `--libReplacement`를 대신 사용하는 것을 고려하십시오.

[여기서 더 보기](https://github.com/microsoft/TypeScript/issues/62209) 및 [해당 pull request](https://github.com/microsoft/TypeScript/pull/62435).

## `tsconfig.json`이 존재할 때 명령줄 파일 지정은 이제 오류입니다

현재 `tsconfig.json`이 있는 폴더에서 `tsc foo.ts`를 실행하면 설정 파일이 완전히 무시됩니다. 입력 파일에 검사 및 방출 옵션이 적용될 것으로 예상했다면 이는 종종 매우 혼란스러웠습니다.

TypeScript 6.0에서는 `tsconfig.json`을 포함하는 디렉토리에서 파일 인자와 함께 `tsc`를 실행하면 이 동작을 명확히 하기 위해 오류가 발생합니다:

```
error TS5112: tsconfig.json이 존재하지만 파일이 명령줄에 지정된 경우 로드되지 않습니다. 이 오류를 건너뛰려면 '--ignoreConfig'를 사용하십시오.

```

`tsconfig.json`을 무시하고 TypeScript 기본값으로 `foo.ts`만 컴파일하고 싶다면, 새로운 `--ignoreConfig` 플래그를 사용할 수 있습니다.

```
tsc --ignoreConfig foo.ts

```

[이 이슈에서 더 보기](https://github.com/microsoft/TypeScript/issues/62197) 및 해당 [pull request](https://github.com/microsoft/TypeScript/pull/62477).

# TypeScript 7.0 준비

TypeScript 6.0은 전환 릴리스로 설계되었습니다. TypeScript 6.0에서 사용 중단된 옵션은 `"ignoreDeprecations": "6.0"`이 설정된 경우에도 오류 없이 계속 작동하지만, 해당 옵션은 **TypeScript 7.0 (네이티브 TypeScript 포트)에서 완전히 제거될 것입니다**. TypeScript 6.0 업그레이드 후 사용 중단 경고가 표시되는 경우, TypeScript 7.0 (또는 [네이티브 미리보기](https://www.npmjs.com/package/@typescript/native-preview))을 프로젝트에 채택하기 전에 해결하는 것이 좋습니다.

# 다음 단계는 무엇인가요?

이제 TypeScript 6.0이 npm에 출시되었으므로, 팀은 TypeScript 7.0을 안정화하는 데 집중할 것입니다. 이는 들리는 것보다 훨씬 가깝습니다. 몇 달 안에 릴리스될 것으로 예상되며, 이미 Microsoft 내부 및 외부에서 매우 큰 코드베이스에 광범위하게 채택되고 있습니다. 따라서 팀이 TypeScript 7.0의 [npm 네이티브 미리보기](https://www.npmjs.com/package/@typescript/native-preview)와 [VS Code 확장 프로그램](https://marketplace.visualstudio.com/items?itemName=TypeScriptTeam.native-preview)의 야간 빌드를 사용해 보도록 권장합니다. TypeScript 7.0에 대한 피드백은 큰 도움이 될 것이며, [이슈 트래커에 이슈를 제출할 수 있습니다](https://github.com/microsoft/typescript-go/issues).

하지만 TypeScript 6.0은 오늘날 채택할 수 있는 안정적인 릴리스이며, 즉시 사용할 수 있는 여러 개선 사항과 새로운 기능을 포함하고 있습니다. 이 릴리스가 모두에게 원활한 전환이 되기를 바라며, 여러분의 경험에 대해 듣기를 기대합니다.
