> 원문: [Giving V8 a Heads-Up: Faster JavaScript Startup with Explicit Compile Hints](https://v8.dev/blog/explicit-compile-hints) — V8 Blog

빠른 JavaScript 실행은 반응성이 뛰어난 웹 앱의 핵심입니다. V8의 고급 최적화 기능에도 불구하고, 시작 시 중요한 JavaScript를 파싱하고 컴파일하는 과정은 성능 병목 현상을 일으킬 수 있습니다. 초기 스크립트 컴파일 중에 어떤 JavaScript 함수를 컴파일할지 미리 알면 웹 페이지 로딩 속도를 높일 수 있습니다.

네트워크에서 로드된 스크립트를 처리할 때 V8은 각 함수에 대해 즉시 컴파일("eagerly")할지 아니면 이 프로세스를 연기할지 결정해야 합니다. 아직 컴파일되지 않은 함수가 나중에 호출되면 V8은 필요에 따라 컴파일해야 합니다.

JavaScript 함수가 페이지 로딩 중에 호출된다면, 이를 즉시 컴파일하는 것이 이롭습니다. 왜냐하면:

*   스크립트의 초기 처리 중에 함수 종료를 찾기 위해 최소한 경량 파싱은 수행해야 합니다. JavaScript에서 함수 종료를 찾는 것은 전체 문법을 파싱해야 합니다 (중괄호 개수를 세는 바로가기는 없습니다. 문법이 너무 복잡합니다). 먼저 경량 파싱을 수행하고 나중에 실제 파싱을 하는 것은 중복 작업입니다.
*   함수를 즉시 컴파일하기로 결정하면, 작업이 백그라운드 스레드에서 수행되며, 네트워크에서 스크립트를 로드하는 작업과 일부가 겹칩니다. 대신 함수가 호출될 때만 컴파일하면, 메인 스레드가 함수 컴파일 전까지 진행할 수 없기 때문에 작업을 병렬화하기에는 너무 늦습니다.

V8이 JavaScript를 파싱하고 컴파일하는 방법에 대해 더 자세히 알아보려면 [여기](https://v8.dev/blog/preparser)를 참조하십시오.

많은 웹 페이지는 즉시 컴파일할 올바른 함수를 선택하는 것으로 이점을 얻을 것입니다. 예를 들어, 인기 있는 웹 페이지를 대상으로 한 저희 실험에서 20개 중 17개에서 성능 향상을 보였으며, 평균 포그라운드 파스 및 컴파일 시간 감소는 630ms였습니다.

저희는 웹 개발자가 어떤 JavaScript 파일과 함수가 즉시 컴파일될지 제어할 수 있도록 하는 기능인 [Explicit Compile Hints](https://github.com/WICG/explicit-javascript-compile-hints-file-based)를 개발하고 있습니다. Chrome 136부터는 개별 파일을 즉시 컴파일하도록 선택할 수 있는 버전이 배포되고 있습니다.

이 버전은 "핵심 파일"이 있어서 즉시 컴파일 대상으로 선택하거나, 소스 파일 간에 코드를 이동하여 이러한 핵심 파일을 만들 수 있는 경우 특히 유용합니다.

파일 상단에 다음 매직 주석을 삽입하여 전체 파일에 대해 즉시 컴파일을 트리거할 수 있습니다.

```
//# allFunctionsCalledOnLoad
```

하지만 이 기능은 신중하게 사용해야 합니다. 너무 많이 컴파일하면 시간과 메모리를 소비하게 됩니다!

# 직접 확인해 보세요 - 컴파일 힌트 작동 방식 [#](#see-for-yourself---compile-hints-in-action)

함수 이벤트를 기록하도록 v8에 지시하여 컴파일 힌트가 작동하는 것을 관찰할 수 있습니다. 예를 들어, 최소한의 테스트를 설정하기 위해 다음 파일을 사용할 수 있습니다.

index.html:

```
<script src="script1.js"></script><script src="script2.js"></script>
```

script1.js:

```
function testfunc1() {  console.log('testfunc1 called!');}testfunc1();
```

script2.js:

```
//# allFunctionsCalledOnLoadfunction testfunc2() {  console.log('testfunc2 called!');}testfunc2();
```

코드 캐싱이 실험을 방해하지 않도록 Chrome을 깨끗한 사용자 데이터 디렉토리와 함께 실행해야 합니다. 예시 명령줄은 다음과 같습니다.

```
rm -rf /tmp/chromedata && google-chrome --no-first-run --user-data-dir=/tmp/chromedata --js-flags=--log-function_events > log.txt
```

테스트 페이지로 이동한 후 로그에서 다음 함수 이벤트를 볼 수 있습니다.

```
$ grep testfunc log.txtfunction,preparse-no-resolution,5,18,60,0.036,179993,testfunc1function,full-parse,5,18,60,0.003,181178,testfunc1function,parse-function,5,18,60,0.014,181186,testfunc1function,interpreter,5,18,60,0.005,181205,testfunc1function,full-parse,6,48,90,0.005,184024,testfunc2function,interpreter,6,48,90,0.005,184822,testfunc2
```

`testfunc1`은 지연 컴파일되었기 때문에 나중에 호출될 때 `parse-function` 이벤트를 볼 수 있습니다.

```
function,parse-function,5,18,60,0.014,181186,testfunc1
```

`testfunc2`의 경우, 컴파일 힌트가 즉시 파싱 및 컴파일을 강제했기 때문에 해당 이벤트가 보이지 않습니다.

# Explicit Compile Hints의 미래 [#](#future-of-explicit-compile-hints)

장기적으로는 개별 함수를 즉시 컴파일 대상으로 선택하는 방향으로 나아가고자 합니다. 이를 통해 웹 개발자는 컴파일할 함수를 정확하게 제어하고, 웹 페이지를 최적화하기 위해 컴파일 성능의 마지막 부분을 짜낼 수 있습니다. 계속 지켜봐 주세요!
