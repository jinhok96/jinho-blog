> 원문: [How we made JSON.stringify more than twice as fast](https://v8.dev/blog/json-stringify) — V8 Blog

`JSON.stringify`는 데이터를 직렬화하는 핵심 JavaScript 함수입니다. 이 함수의 성능은 네트워크 요청을 위한 데이터 직렬화부터 `localStorage`에 데이터를 저장하는 일반적인 웹 작업에 직접적인 영향을 미칩니다. 더 빠른 `JSON.stringify`는 더 빠른 페이지 상호 작용과 더 반응적인 애플리케이션으로 이어집니다. 따라서 최근 엔지니어링 노력을 통해 V8의 `JSON.stringify` 성능을 **2배 이상 향상**시켰다는 소식을 전하게 되어 기쁩니다. 이 게시물에서는 이러한 성능 개선을 가능하게 한 기술적 최적화를 자세히 살펴보겠습니다.

# 부작용 없는 고속 경로 [#](#a-side-effect-free-fast-path)

이 최적화의 기반은 간단한 전제에 기반한 새로운 고속 경로입니다. 객체 직렬화가 부작용을 일으키지 않는다고 보장할 수 있다면, 훨씬 빠르고 특화된 구현을 사용할 수 있습니다. 이 맥락에서 "부작용"이란 객체의 단순하고 간소화된 순회를 방해하는 모든 것을 의미합니다.
여기에는 직렬화 중 사용자 정의 코드 실행과 같은 명백한 경우뿐만 아니라 가비지 컬렉션 주기를 트리거할 수 있는 더 미묘한 내부 작업도 포함됩니다. 부작용을 정확히 유발할 수 있는 것과 이를 방지하는 방법에 대한 자세한 내용은 [Limitations](#limitations)를 참조하십시오.

V8이 직렬화에서 이러한 효과가 없음을 확인할 수 있는 한, 이 고도로 최적화된 경로를 유지할 수 있습니다. 이를 통해 범용 직렬화 프로그램에 필요한 많은 비용이 많이 드는 검사 및 방어 논리를 우회할 수 있으며, 일반적인 순수 데이터 JavaScript 객체 유형에 대해 상당한 속도 향상을 얻을 수 있습니다.

또한 새로운 고속 경로는 재귀적인 범용 직렬화와 달리 반복적입니다. 이러한 아키텍처 선택은 스택 오버플로 검사의 필요성을 제거하고 [다른 문자열 표현 처리](#handling-different-string-representations) 후 신속하게 재개할 수 있도록 할 뿐만 아니라 개발자가 이전에 가능했던 것보다 훨씬 더 깊은 중첩 객체 그래프를 직렬화할 수 있도록 합니다.

# 다른 문자열 표현 처리 [#](#handling-different-string-representations)

V8의 문자열은 1바이트 또는 2바이트 문자로 표현될 수 있습니다. 문자열이 ASCII 문자만 포함하는 경우 V8에서 1바이트당 1바이트를 사용하는 1바이트 문자열로 저장됩니다. 그러나 문자열에 ASCII 범위를 벗어나는 단일 문자만 포함되어 있으면 문자열의 모든 문자가 2바이트 표현을 사용하며 메모리 사용량이 본질적으로 두 배가 됩니다.

통합 구현의 지속적인 분기 및 유형 검사를 피하기 위해 전체 문자열 처리기가 이제 문자 유형에 대한 템플릿으로 사용됩니다. 이는 두 개의 별개이고 특화된 버전의 직렬화 프로그램을 컴파일한다는 것을 의미합니다. 하나는 1바이트 문자열에 대해 완전히 최적화되어 있고 다른 하나는 2바이트 문자열에 대해 최적화되어 있습니다. 이는 이진 크기에 영향을 미치지만 증가된 성능은 확실히 그럴 가치가 있다고 생각합니다.

구현은 혼합 인코딩을 효율적으로 처리합니다. 직렬화 중에는 고속 경로에서 처리할 수 없는 표현(예: 평탄화 중에 GC를 트리거할 수 있는 [`ConsString`](https://crsrc.org/c/v8/src/objects/string.h;drc=9768251f3e8f598d82420259a940d2057ed56b42;l=1013))을 감지하기 위해 각 문자열의 인스턴스 유형을 검사해야 하므로 느린 경로로 대체해야 합니다. 이 필수 검사는 문자열이 1바이트 또는 2바이트 인코딩을 사용하는지 여부도 드러냅니다.

따라서 낙관적인 1바이트 문자열 처리기에서 2바이트 버전으로 전환하는 결정은 본질적으로 무료입니다. 이 기존 검사에서 2바이트 문자열이 발견되면 새 2바이트 문자열 처리기가 생성되고 현재 상태를 상속합니다. 마지막으로 초기 1바이트 문자열 처리기의 출력과 2바이트 문자열 처리기의 출력을 단순히 연결하여 최종 결과가 구성됩니다. 이 전략은 일반적인 경우에 고도로 최적화된 경로를 유지하도록 보장하며, 2바이트 문자를 처리하기 위한 전환은 가볍고 효율적입니다.

# SIMD를 사용한 문자열 직렬화 최적화 [#](#optimizing-string-serialization-with-simd)

JSON으로 직렬화할 때 이스케이프해야 하는 문자(`"` 또는 `\` 등)를 JavaScript의 모든 문자열에 포함할 수 있습니다. 이를 찾기 위한 기존의 문자별 루프는 느립니다.

이를 가속화하기 위해 문자열 길이에 기반한 2단계 전략을 사용합니다.

*   더 긴 문자열의 경우 전용 하드웨어 SIMD 명령어(예: ARM64 Neon)로 전환합니다. 이를 통해 훨씬 더 큰 문자열 조각을 넓은 SIMD 레지스터로 로드하고 몇 개의 명령만으로 한 번에 이스케이프 가능한 문자가 있는지 여러 바이트를 확인할 수 있습니다. ([source](https://crsrc.org/c/v8/src/json/json-stringifier.cc;drc=1645281bbd1b183a252835d376166bd210135bbe;l=3369))
*   하드웨어 명령어 설정 비용이 너무 높은 짧은 문자열의 경우 SWAR(SIMD Within A Register)이라는 기술을 사용합니다. 이 접근 방식은 표준 범용 레지스터에서 정교한 비트 연산 논리를 사용하여 매우 낮은 오버헤드로 한 번에 여러 문자를 처리합니다. ([source](https://crsrc.org/c/v8/src/json/json-stringifier.cc;drc=1645281bbd1b183a252835d376166bd210135bbe;l=3353))

방법에 관계없이 프로세스는 매우 효율적입니다. 문자열을 청크별로 빠르게 스캔합니다. 청크에 특수 문자가 없으면(일반적인 경우) 전체 문자열을 복사할 수 있습니다.

# 고속 경로의 고속 차선 [#](#the-express-lane-on-the-fast-path)

주요 고속 경로 내에서도 더 빠르고 '고속 차선'을 위한 기회를 발견했습니다. 기본적으로 고속 경로는 객체의 속성을 반복하고 각 키에 대해 일련의 검사를 수행해야 합니다. 키가 `Symbol`이 아닌지 확인하고, 열거 가능한지 확인하고, 마지막으로 이스케이프해야 하는 문자(예: `"` 또는 `\`)에 대해 문자열을 스캔합니다.

이를 제거하기 위해 객체의 [hidden class](https://v8.dev/docs/hidden-classes)에 플래그를 도입했습니다. 객체의 모든 속성을 직렬화한 후, 속성 키가 `Symbol`이 아니고, 모든 속성이 열거 가능하며, 속성 키에 이스케이프해야 하는 문자가 포함되지 않은 경우 hidden class를 fast-json-iterable로 표시합니다.

이전에 직렬화한 객체와 동일한 hidden class를 가진 객체(예: 모양이 같은 객체 배열)를 직렬화하고 fast-json-iterable인 경우 추가 검사 없이 모든 키를 문자열 버퍼로 복사할 수 있습니다.

배열의 객체는 종종 동일한 hidden class를 가지므로, `JSON.parse`에도 이 최적화를 추가하여 배열을 파싱하는 동안 빠른 키 비교에 활용할 수 있습니다.

# 더 빠른 double-to-string 알고리즘 [#](#a-faster-double-to-string-algorithm)

숫자를 문자열 표현으로 변환하는 것은 놀라울 정도로 복잡하고 성능에 중요한 작업입니다. `JSON.stringify`에 대한 작업을 부분으로, 짧은 길이의 숫자-문자열 변환을 위해 Grisu3 알고리즘을 [Dragonbox](https://github.com/jk-jeon/dragonbox)로 교체하여 이 프로세스를 크게 가속화할 수 있는 기회를 파악했습니다.

이 최적화는 `JSON.stringify` 프로파일링에 의해 주도되었지만, 새로운 Dragonbox 구현은 V8의 모든 `Number.prototype.toString()` 호출에 혜택을 줍니다. 즉, JSON 직렬화뿐만 아니라 숫자를 문자열로 변환하는 모든 코드가 이 성능 향상을 무료로 볼 수 있습니다.

# 기본 임시 버퍼 최적화 [#](#optimizing-the-underlying-temporary-buffer)

모든 문자열 빌드 작업에서 상당한 오버헤드 소스는 메모리 관리 방법입니다. 이전에 문자열 처리기는 C++ 힙에 있는 단일 연속 버퍼에 출력을 빌드했습니다. 간단하지만 이 접근 방식에는 상당한 단점이 있습니다. 버퍼 공간이 부족할 때마다 더 큰 버퍼를 할당하고 기존 전체 내용을 복사해야 했습니다. 대규모 JSON 객체의 경우 다시 할당 및 복사 주기는 주요 성능 오버헤드를 발생시켰습니다.

중요한 통찰력은 이 임시 버퍼를 연속으로 강제하는 것이 실제 이점이 없다는 것입니다. 최종 결과는 매우 마지막에 단일 문자열로만 조립되기 때문입니다.

이를 염두에 두고 기존 시스템을 세그먼트 버퍼로 교체했습니다. 하나의 크고 성장하는 메모리 블록 대신, 이제 V8의 Zone 메모리에 할당된 작은 버퍼(또는 "세그먼트") 목록을 사용합니다. 세그먼트가 가득 차면 새 세그먼트를 할당하고 거기에 계속 쓰면 비싼 복사 작업이 완전히 제거됩니다.

# Limitations [#](#limitations)

새로운 고속 경로는 일반적이고 간단한 경우를 전문화하여 속도를 달성합니다. 직렬화되는 데이터가 이러한 기준을 충족하지 않으면 V8은 정확성을 보장하기 위해 범용 직렬화 프로그램으로 대체됩니다. 전체 성능 이점을 얻으려면 `JSON.stringify` 호출이 다음 조건을 준수해야 합니다.

*   `replacer` 또는 `space` 인수 없음: `replacer` 함수 또는 예쁘게 인쇄하기 위한 `space`/`gap` 인수를 제공하는 것은 범용 경로에서만 처리되는 기능입니다. 고속 경로는 압축되고 변환되지 않은 직렬화를 위해 설계되었습니다.
*   일반 데이터 객체 및 배열: 직렬화되는 객체는 간단한 데이터 컨테이너여야 합니다. 이는 객체와 해당 프로토타입에 사용자 정의 `.toJSON()` 메서드가 없어야 함을 의미합니다. 고속 경로는 사용자 정의 직렬화 로직이 없는 표준 프로토타입(예: `Object.prototype` 또는 `Array.prototype`)을 가정합니다.
*   객체의 인덱스 속성 없음: 고속 경로는 일반 문자열 기반 키가 있는 객체에 대해 최적화되어 있습니다. 객체에 배열과 같은 인덱스 속성('0', '1' 등)이 포함된 경우 더 느리고 일반적인 직렬화 프로그램에서 처리됩니다.
*   간단한 문자열 유형: 일부 내부 V8 문자열 표현(예: `ConsString`)은 직렬화하기 전에 평탄화하기 위해 메모리 할당이 필요할 수 있습니다. 고속 경로는 이러한 할당을 트리거할 수 있는 작업을 피하며 간단한 순차 문자열에 가장 적합합니다. 이것은 웹 개발자가 영향을 미치기 어려운 것입니다. 하지만 대부분의 경우 잘 작동해야 합니다.

API 응답을 위한 데이터 직렬화 또는 구성 객체 캐싱과 같은 대부분의 사용 사례에서는 이러한 조건이 자연스럽게 충족되어 개발자가 성능 향상을 자동으로 활용할 수 있습니다.

# Conclusion [#](#conclusion)

`JSON.stringify`를 최상위 논리부터 핵심 메모리 및 문자 처리 작업에 이르기까지 처음부터 다시 생각함으로써 JetStream2 json-stringify-inspector 벤치마크에서 측정된 2배 이상의 성능 향상을 제공했습니다. 다른 플랫폼의 결과는 아래 그림을 참조하십시오. 이러한 최적화는 V8 버전 13.8(Chrome 138)부터 사용할 수 있습니다.
